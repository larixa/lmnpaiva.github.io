1.1 O que é o tenant Microsoft Entra ID

O tenant é a instância lógica do diretório da organização. Nele vivem os objetos de identidade usados para autenticação, autorização e governança: usuários, grupos, dispositivos, aplicações, service principals, roles, políticas e logs. Pense no tenant como o “plano de controle” de identidade do Microsoft 365, Azure e aplicações integradas.

1.2 Conceitos base: usuário, grupo, dispositivo e aplicação

Usuário representa uma identidade humana. Pode ser cloud-only, sincronizado do AD DS ou externo. Grupo organiza acesso e licenciamento. Pode ser Security group, Microsoft 365 group, assigned ou dynamic. Dispositivo é um objeto usado em SSO, Conditional Access, compliance e device trust. Aplicação pode aparecer como App Registration, Enterprise Application ou service principal, dependendo do contexto.

1.3 Users, groups and license management

Para criar usuários com qualidade, pense em JML: Joiner, Mover e Leaver. O usuário deve ter UPN correto, display name, usage location, department, job title, country, manager e grupos iniciais. Atributos corretos alimentam dynamic groups, access reviews, licença por grupo e relatórios.

Exemplo de lab: crie Ana Martins como Claim Handler, Bruno Costa como Claim Manager, Carla Ribeiro como Service Desk Analyst e Diego Silva como Developer. O nome é da pessoa; o cargo fica no atributo Job title. Evite nomes como “Ana ClaimHandler”.

Group-based licensing aplica licenças por membership. Isso escala melhor que atribuição manual. A pegadinha é Usage location: se não estiver definido, a licença pode falhar.

Regra dinâmica exemplo:
(user.country -eq "PT") and (user.department -eq "Claims") and (user.accountEnabled -eq true)

1.4 Administrative Units e custom security attributes

Administrative Units reduzem escopo administrativo. Em vez de dar User Administrator no tenant inteiro, você cria uma AU como AU-Portugal e atribui Helpdesk Administrator apenas naquele escopo. Isso é um exemplo direto de least privilege.

Custom security attributes são atributos chave-valor definidos pela organização, úteis para ABAC e segmentação. Exemplo: Project = ClaimsPortal, DataClass = Confidential. Eles ajudam quando grupos não são flexíveis o suficiente para representar contexto de negócio.

1.6 RBAC, permissões, funções administrativas e least privilege no Microsoft Entra

RBAC significa Role-Based Access Control. Em termos práticos, é o modelo usado para definir quem recebe uma permissão, qual função essa pessoa ou identidade pode executar, onde essa permissão vale e por quanto tempo ela deve ficar ativa.

Na SC-300, RBAC normalmente aparece em perguntas de cenário. A prova não costuma perguntar apenas “o que é RBAC?”. Ela pergunta algo como: “Qual função deve ser atribuída para resolver o problema com o menor privilégio possível?”

Uma atribuição de função no Microsoft Entra pode ser lida como uma frase simples: “Larissa pode executar tarefas de Helpdesk Administrator apenas nos usuários da Administrative Unit Portugal, durante uma janela aprovada pelo PIM.”

Essa frase contém os quatro elementos que você precisa identificar em questões de prova: principal, função, escopo e duração.

1.6.1 Diferença entre Microsoft Entra RBAC e Azure RBAC

Esse é um ponto que confunde muita gente. Microsoft Entra RBAC administra objetos do diretório, como usuários, grupos, aplicações, Conditional Access, métodos de autenticação e roles. Azure RBAC administra recursos do Azure, como subscriptions, resource groups, VMs, Storage Accounts e Key Vault.

Se fala sobre VM, storage, Key Vault, subscription ou resource group, pense em Azure RBAC. Se fala sobre acesso temporário, aprovação e just-in-time, pense em PIM.

1.6.2 Conceito de least privilege

Least privilege significa conceder somente a permissão necessária para executar uma tarefa, pelo menor tempo possível e no menor escopo possível. Na prática, você deve evitar três erros: dar uma role poderosa demais, aplicar no tenant inteiro quando uma Administrative Unit resolveria, ou deixar a permissão permanente quando PIM poderia fazer a elevação temporária.

1.6.4 Authentication Administrator vs Privileged Authentication Administrator

Essa diferença cai bastante porque parece pequena, mas muda totalmente a resposta. O Authentication Administrator ajuda usuários comuns com MFA, TAP e informações de segurança. O Privileged Authentication Administrator pode fazer isso também para contas privilegiadas, como admins.

1.6.5 Funções personalizadas no Microsoft Entra

Uma função personalizada é usada quando nenhuma função interna oferece exatamente o conjunto de permissões necessário. Ela é composta por uma definição de função (role definition), que descreve as ações permitidas, e por uma atribuição de função (role assignment), que aplica essa definição a um principal dentro de um escopo.

Em vez de dar Application Administrator, você pode criar uma função personalizada com permissões específicas relacionadas a credenciais de aplicações.

Como implementar uma custom role no tenant lab:
1. Acesse Microsoft Entra admin center → Roles & admins → New custom role.
2. Dê um nome claro, por exemplo APP-Credential-Rotation-Operator.
3. Adicione permissões relacionadas à leitura de aplicações e atualização de credenciais.
4. Atribua a role a um grupo, não diretamente a usuários, sempre que possível.
5. Limite o escopo ao tenant, a uma application ou a um conjunto controlado, se o portal permitir o escopo desejado.
6. Se disponível, torne a atribuição eligible via PIM em vez de permanente.

2. Lifecycle de usuário: criação, atributos, licenciamento e desativação

Um usuário não é apenas um nome e um e-mail. Em ambientes reais, atributos como department, job title, country, manager, usage location e employee type direcionam licenciamento, grupos dinâmicos, delegação por Administrative Unit, access reviews e políticas de acesso.

  2. Definir Usage location. Sem esse atributo, a atribuição de licenças Microsoft 365 pode falhar, mesmo que o usuário esteja no grupo correto.
  3. Adicionar o usuário aos grupos corretos, preferencialmente por grupos dinâmicos quando a regra for baseada em atributo estável.
  4. Atribuir licença por grupo, não manualmente usuário por usuário. Isso reduz erro operacional e facilita auditoria.
  5. Definir manager. Esse campo é importante para workflows, access reviews, aprovação de access packages e Lifecycle Workflows.
  6. Na saída do usuário, bloquear sign-in, revogar sessões, remover grupos, converter/encaminhar mailbox quando aplicável e revisar owners de grupos/apps.

Password Hash Synchronization sincroniza um hash derivado da senha. Não envia senha em texto claro. É simples, resiliente e permite autenticação cloud mesmo se o ambiente on-prem estiver indisponível.

Pass-through Authentication mantém validação da senha no AD DS local por meio de agentes. É útil quando a empresa precisa aplicar algumas regras on-prem no momento do login, mas depende da disponibilidade dos agentes.

Federation/AD FS redireciona o usuário para um serviço federado. Ainda aparece em ambientes legados ou requisitos específicos, mas para muitos cenários Microsoft recomenda reduzir complexidade migrando para cloud authentication.

Seamless SSO não é método de autenticação isolado. Ele complementa PHS ou PTA para reduzir prompts em dispositivos corporativos na rede, usando Kerberos e a conta AZUREADSSOACC.

5. Cross-tenant synchronization explicado com cuidado

Cross-tenant synchronization copia/provisiona usuários de um tenant de origem para um tenant de destino usando o mecanismo de provisionamento do Microsoft Entra. Ele não “move” a conta original. O usuário continua existindo no tenant de origem, mas um objeto correspondente é criado e mantido no tenant de destino.

  1. No tenant de destino, configure cross-tenant access settings para confiar no tenant de origem e permitir inbound synchronization.
  2. No tenant de origem, crie/configure a Enterprise Application de cross-tenant synchronization.
  3. Defina o escopo: quais usuários ou grupos serão sincronizados.
  4. Mapeie atributos: displayName, mail, userPrincipalName, department, manager e outros atributos necessários.
  5. Teste o provisionamento com poucos usuários antes de expandir.
Estudo de caso: A Contoso comprou a Fabrikam, mas os tenants ainda não serão consolidados. Cross-tenant synchronization é mais adequado porque automatiza criação, atualização e remoção no tenant de destino.

3. Temporary Access Pass — TAP

TAP é uma senha temporária emitida por administrador para permitir onboarding ou recuperação segura quando o usuário ainda não tem método forte. Ele deve ser temporário, auditável e usado para registrar um método definitivo, como FIDO2 ou Authenticator.

Use TAP para

  • Onboarding de novo usuário.
  • Recuperação quando o telefone com Authenticator foi perdido.
  • Registrar passkey ou WHfB sem senha inicial fraca.

Não use TAP como

  • Bypass permanente de MFA.
  • Solução para todos os bloqueios de acesso.
  • Alternativa a corrigir Conditional Access mal desenhado.

4. SSPR e password writeback

Self-Service Password Reset permite que usuários redefinam a própria senha. Em ambiente híbrido, password writeback grava a nova senha de volta no AD DS on-premises. Isso evita abrir tickets simples no Service Desk e melhora a experiência do usuário.

  1. Habilite SSPR para um grupo piloto, não para todos de uma vez.
  2. Configure dois métodos de verificação para reset, quando o risco exigir.
  3. Em ambiente híbrido, habilite password writeback no Entra Connect/Cloud Sync conforme suportado.
  4. Teste com uma conta não administrativa e valide se a senha sincronizou para on-prem/cloud.

5. Password Protection

Microsoft Entra Password Protection bloqueia senhas fracas e termos customizados, como o nome da empresa, produto interno ou padrões previsíveis. Em ambiente híbrido, agentes nos Domain Controllers aplicam a proteção também em resets locais.

Estudo de caso: A Contoso percebeu tentativas de password spray usando variações de “Contoso2026!”. A resposta correta não é apenas “forçar MFA”. MFA ajuda, mas Password Protection com banned passwords customizadas reduz a chance de usuários escolherem senhas previsíveis no próximo reset.

ID Protection avalia sign-in risk e user risk. Sign-in risk se refere à tentativa atual: anonymous IP, impossible travel, unfamiliar properties, password spray etc. User risk indica que a identidade como um todo pode estar comprometida, como leaked credentials.

2. Delegated permissions vs Application permissions

Regra de prova: se o cenário diz “sem usuário interativo”, “daemon”, “background job” ou “scheduled task”, pense em application permission, service principal ou managed identity. Se diz “em nome do usuário”, pense em delegated permission.

Checklist de prontidão para a SC-300

Use isto na última semana. Marque mentalmente o que você consegue explicar sem consultar o material.

  • Explicar tenant, user, group, device, app object e service principal.
  • Diferenciar Entra registered, Entra joined e Hybrid joined.
  • Criar usuários, grupos dinâmicos, AUs e licenças por grupo.
  • Explicar PHS, PTA, Federation, Seamless SSO e Kerberos.
  • Diferenciar B2B Collaboration, Direct Connect, B2C e Cross-tenant sync.
  • Configurar MFA, TAP, FIDO2, WHfB, CBA e SSPR.
  • Criar políticas Conditional Access em report-only e testar com What If.
  • Explicar sign-in risk vs user risk e remediação adequada.
  • Diferenciar SAML, OIDC/OAuth, App Registration e Enterprise App.
  • Explicar Application Proxy, connectors e KCD.
  • Usar Managed Identity para acessar Key Vault sem segredo.
  • Configurar PIM, Access Reviews e Access Packages.
  • Enviar logs para Log Analytics e fazer KQL básico.
  • Explicar Zero Trust aplicado à identidade.
  • Explicar GSA Internet Access, Private Access, client e traffic profiles.
  • Responder questões sempre buscando menor privilégio e menor impacto.
