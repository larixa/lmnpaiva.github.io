Domínio 1 · 20-25%

Implement and manage user identities

Ciclo de vida das identidades humanas: tenant, roles, usuários, grupos, devices, licenças, externos e identidade híbrida.

1.1 Configure and manage a Microsoft Entra tenant

Conceito: O tenant é a instância lógica da organização. Administrá-lo é definir limites de segurança, delegação, domínios e configurações globais.

  • Built-in roles: Global Administrator, User Administrator, Helpdesk Administrator, Application Administrator — use o mais específico.
  • Custom roles: quando roles padrão dão permissões demais ou de menos.
  • Administrative Units: escopam administração para subconjuntos de usuários/grupos/devices por região ou departamento.
  • Effective permissions: pode vir de role direta, grupo role-assignable, PIM, AU ou combinação.
Exam tip:

Delegação por departamento/filial → Administrative Units. Permissão mínima para uma tarefa → role específico + least privilege.

Atenção:

Evite Global Administrator para tarefas rotineiras. Global Admin deve ser raro, protegido por PIM, MFA forte e contas break-glass separadas.

1.2 Identidades, grupos e licenças

  • Grupos dinâmicos: membership por regra de atributo como department, jobTitle.
  • Grupos assigned: controle manual, bom para acesso sensível.
  • Licenciamento por grupo: reduz esforço manual, monitorar conflitos.
  • Custom security attributes: atributos chave-valor para ABAC e classificação.
  • Bulk operations: CSV/PowerShell para onboarding/offboarding em massa.
Exam tip:

Atributos confiáveis + regra clara → dynamic group melhor que adicionar manualmente.

1.3 External users, B2B e cross-tenant

  • B2B collaboration: cria objeto guest no seu tenant.
  • Cross-tenant access settings: controles inbound/outbound por parceiro, MFA trust, device claims.
  • Cross-tenant synchronization: automatiza criação/atualização de usuários B2B entre tenants da mesma org.
  • External collaboration settings: regras gerais de convite, restrições de domínio.
Exam tip:

Colaboração entre dois tenants com controles específicos por parceiro → cross-tenant access settings. Convite genérico → external collaboration settings. Manter guests atualizados automaticamente → cross-tenant synchronization.

1.4 Hybrid identity: PHS, PTA, Federation

  • Password Hash Sync (PHS): hash derivado sincronizado para Entra. Simples, resiliente, autenticação na nuvem. Recomendado em muitos cenários modernos.
  • Pass-through Auth (PTA): senha validada em tempo real contra o AD DS. Depende de agentes on-premises disponíveis.
  • Federation / AD FS: autenticação delegada a provedor federado. Maior complexidade, casos específicos como smart cards legados.
  • Microsoft Entra Connect Sync: sincronização tradicional e rica para cenários complexos.
  • Cloud Sync: agente mais leve, útil para múltiplas florestas e cenários simples.
Exam tip:

Menor complexidade + maior resiliência → PHS. Validar senha no AD DS → PTA. AD FS / claims complexas / smart cards → Federation.

Domínio 2 · 25-30%

Implement authentication and access management

Métodos de autenticação, MFA, passwordless, Conditional Access, risco e Global Secure Access.

2.1 User authentication

  • MFA: fator extra além da senha. Exigido por CA, Security Defaults ou configurações específicas.
  • Microsoft Authenticator: push, number matching e métodos passwordless.
  • FIDO2/Passkeys: criptografia assimétrica, resistente a phishing.
  • Certificate-based authentication: útil em organizações com PKI e requisitos fortes.
  • Temporary Access Pass: credencial temporária para bootstrap passwordless ou recuperação.
  • SSPR: self-service password reset com métodos de autenticação registrados.
  • Windows Hello for Business: autenticação forte vinculada ao dispositivo com PIN/biometria.
Exam tip:

Diferença crucial: permitir método de autenticação (Authentication methods policy) é diferente de exigir método em contexto (Conditional Access).

2.2 Conditional Access

Conceito: Motor de decisão "se isso acontecer, aplique este controle". Usa sinais de usuário, app, localização, device, risco e plataforma.

  • Assignments: usuários/grupos, cloud apps e actions.
  • Conditions: sign-in risk, user risk, device platform, locations, client apps.
  • Grant controls: bloquear, exigir MFA, compliant device, hybrid joined.
  • Session controls: sign-in frequency, persistent browser session, App Control via Defender.
  • Report-only: avalie impacto antes de enforcement.
  • What If tool: simula como políticas se aplicam a determinado cenário.
  • Authentication context: aplica controles a ações sensíveis específicas dentro de apps.
Atenção:

CA mal configurado pode bloquear o tenant. Mantenha contas break-glass excluídas de políticas restritivas e monitoradas.

2.3 Microsoft Entra ID Protection e risco

  • User risk: probabilidade de a identidade estar comprometida.
  • Sign-in risk: risco daquela tentativa específica de login.
  • Risky sign-ins: impossible travel, anonymous IP, leaked credentials.
  • CAE (Continuous Access Evaluation): revoga acesso mais rapidamente a eventos críticos sem esperar expiração do token.
Exam tip:

Risco calculado automaticamente + resposta baseada em risco → ID Protection + CA. Encerrar acesso rapidamente após evento crítico → CAE ou revogar sessões.

2.4 Global Secure Access (GSA)

Conceito: Microsoft Entra Internet Access + Microsoft Entra Private Access. Zero Trust: verificar explicitamente, menor privilégio, assumir violação.

Domínio 3 · 20-25%

Plan and implement workload identities

Identidades para aplicações, automações e recursos Azure — críticas por terem permissões amplas sem usuário humano.

3.1 Managed identity vs service principal vs app registration

Conceito: A escolha correta reduz segredo exposto. Para recurso Azure acessando outro recurso, prefira managed identity.

IdentidadeQuando usarVantagem principal
System-assigned managed identityRecurso Azure acessa outro Azure (VM, Function, App Service)Lifecycle ligado ao recurso, sem segredo para gerir
User-assigned managed identityMúltiplos recursos partilham a mesma identidadeLifecycle independente, reutilizável
Service principalApp multi-tenant, automações, integrações externasControle granular, multi-tenant
App registrationDefinição da aplicação na plataforma de identidadeClient ID, redirect URIs, credentials, API permissions
Exam tip:

Recurso Azure acessa outro Azure → managed identity. App externa/multi-tenant → app registration + service principal. Nunca use user account para automação.

3.2 Enterprise apps, consentimento e SCIM

  • Assignment required: só usuários/grupos atribuídos acessam.
  • App roles: papéis emitidos no token para autorização dentro da app.
  • SCIM: provisioning automático de usuários em SaaS.
  • Delegated permission: app age em nome do usuário.
  • Application permission: app age como ela mesma — normalmente exige admin consent.
  • Admin consent workflow: usuários solicitam aprovação para apps que exigem consentimento admin.
  • Client secrets vs certificados: certificados são preferíveis — menos risco de vazamento.
Exam tip:

SSO autentica. SCIM provisiona. Não confunda os dois.

3.3 Microsoft Entra Application Proxy

Conceito: Publica apps web internas para acesso remoto seguro sem abrir inbound firewall. O Private Network Connector usa saída para a cloud.

  • Pre-authentication Microsoft Entra ID: aplica MFA, CA, risco e logs antes do backend — o mais seguro.
  • Passthrough: backend autentica diretamente, sem pre-auth cloud.
  • KCD (Kerberos Constrained Delegation): SSO para apps que usam Windows Authentication.
  • Connector group: agrupa conectores por região ou app. Em produção, ao menos dois para HA.
Exam tip:

App web interna + SSO + MFA via Entra + sem inbound → Application Proxy. Acesso privado mais amplo por IP/FQDN → GSA Private Access. VPN tradicional raramente é a resposta certa na SC-300.

3.4 Azure Key Vault com managed identity

Conceito: Workload usa managed identity para obter token e aceder Key Vault sem segredo no código.

  • Key Vault Secrets User: lê valores de secrets — role mínima para a maioria das workloads.
  • Key Vault Reader: vê metadados, mas não o valor do secret.
  • Key Vault Crypto User: operações criptográficas com keys.
  • Key Vault Administrator: muito amplo — nunca para aplicações comuns.
  • Azure RBAC vs Access Policies: prefira Azure RBAC no permission model do Key Vault.
Exam tip:

VM/Function acessa Key Vault → enable managed identity + assign Key Vault Secrets User no escopo correto. Contributor no recurso NÃO concede acesso ao Key Vault.

Domínio 4 · 20-25%

Plan and automate identity governance

Quem deve ter acesso, por quanto tempo, quem aprova, quem revisa e como removemos acesso quando deixa de ser necessário.

4.1 Entitlement Management

Conceito: Catalogs + access packages + políticas de solicitação, aprovação, expiração e revisão. Acesso governado em vez de manual.

  • Catalog: contêiner lógico de recursos governados.
  • Access package: pacote com recursos e políticas.
  • Connected organizations: organizações externas confiáveis.
  • External user lifecycle: remove guest automaticamente quando não há mais pacotes ativos.
  • Terms of Use: termos que usuários devem aceitar para aceder recursos.
Exam tip:

Acesso sob solicitação + aprovação + expiração + externos → entitlement management / access package.

4.2 Access reviews

Conceito: Revisões periódicas para confirmar se usuários, guests ou admins ainda precisam de acesso.

  • Reviewers: manager para equipes, owner para apps/grupos, self-review em cenários menos críticos.
  • Auto-apply: aplica decisões automaticamente após o fim da review.
  • Recommendations: sistema recomenda negar usuários inativos.
  • Guest reviews: essencial para reduzir externos esquecidos.
Exam tip:

Recertificação + revisão periódica + remover acesso não utilizado → access reviews.

4.3 Privileged Identity Management (PIM)

Conceito: Reduz privilégio permanente. Admins são eligible e ativam apenas quando necessário, com MFA, justificativa e aprovação.

  • Microsoft Entra roles: Global Administrator, Privileged Role Administrator, CA Administrator.
  • Azure resource roles: Owner, Contributor em subscriptions/resource groups.
  • PIM for Groups: membership/ownership privilegiado sob ativação just-in-time.
  • Approval: exige aprovador para roles de maior risco.
  • Break-glass accounts: fora do PIM, altamente protegidas, monitoradas com alertas de sign-in.
Exam tip:

Reduzir admin permanente mantendo capacidade operacional → PIM. Contas break-glass devem ser excluídas de CA restritivo e monitoradas.

4.4 Logs, Diagnostic settings e KQL

  • Sign-in logs: autenticações, resultado, CA aplicada, risco, IP, device.
  • Audit logs: mudanças administrativas, criação de objetos, alterações em policies e roles.
  • AADProvisioningLogs: provisioning para apps, HR-driven, cross-tenant sync.
  • ServicePrincipalSignInLogs: autenticações de service principals.
  • ManagedIdentitySignInLogs: autenticações de managed identities.
  • Diagnostic settings → Log Analytics Workspace: para retenção, KQL, workbooks e alertas.
Exam tip:

Consulta avançada + retenção + dashboard + correlação → Diagnostic settings → Log Analytics → KQL. Sign-in log individual recente → portal do Entra é suficiente.

KQL essencial

Consultas úteis para identidade

Tabelas para memorizar

  • SigninLogs — login interativo de usuários
  • AuditLogs — mudanças administrativas
  • AADProvisioningLogs — provisioning e sync
  • ServicePrincipalSignInLogs — autenticação de apps
  • ManagedIdentitySignInLogs — autenticação de managed identities

Operadores essenciais

  • where — filtra eventos
  • project — escolhe colunas
  • summarize count() — agrupa e conta
  • order by TimeGenerated desc — eventos recentes primeiro
  • has, contains, in — busca em texto/listas

Falhas de login nas últimas 24h

SigninLogs
| where TimeGenerated > ago(24h)
| where ResultType != 0
| project TimeGenerated, UserPrincipalName, AppDisplayName, ResultType, ResultDescription, IPAddress, ConditionalAccessStatus
| order by TimeGenerated desc

Alterações administrativas recentes

AuditLogs
| where TimeGenerated > ago(7d)
| project TimeGenerated, OperationName, Category, Result, InitiatedBy, TargetResources
| order by TimeGenerated desc

Falhas de service principals

ServicePrincipalSignInLogs
| where TimeGenerated > ago(7d)
| where ResultType != 0
| project TimeGenerated, ServicePrincipalName, AppId, ResourceDisplayName, ResultType, ResultDescription
| order by TimeGenerated desc

Eventos de provisioning com falha

AADProvisioningLogs
| where TimeGenerated > ago(7d)
| where ResultType != "Success"
| project TimeGenerated, Action, ResultType, SourceIdentity, TargetIdentity, ServicePrincipal
| order by TimeGenerated desc
Exam tip:

Quando a pergunta pede consulta avançada, retenção, workbook ou correlação → Diagnostic settings → Log Analytics Workspace → KQL.

  • Dynamic groups: regras de atributo
  • B2B, cross-tenant access settings e cross-tenant sync
  • PHS vs PTA vs Federation — quando usar cada um
  • Connect Sync vs Cloud Sync

Access (D2)

  • MFA, SSPR, TAP, FIDO2, CBA e WHfB
  • Kerberos e Microsoft Entra Kerberos híbrido
  • Conditional Access: assignments, conditions, grant/session
  • ID Protection: user risk vs sign-in risk, CAE
  • GSA: Private, Internet e Microsoft traffic profile

Workloads (D3)

  • Managed identity system-assigned vs user-assigned
  • Key Vault: Key Vault Secrets User + Azure RBAC
  • Enterprise apps, app roles, SSO (SAML/OIDC) e SCIM
  • Application Proxy: pre-auth, connector, KCD
  • Delegated vs application permissions + admin consent

Governance (D4)

  • Entitlement management, catalogs e access packages
  • Access reviews: reviewer, frequência e auto-apply
  • PIM: eligible, ativação, MFA, justificativa, aprovação
  • Break-glass accounts: cloud-only, monitoradas, excluídas de CA
