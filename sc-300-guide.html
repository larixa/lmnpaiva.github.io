1.1 O que é o tenant Microsoft Entra ID

O tenant é a instância lógica do diretório da organização. Nele vivem os objetos de identidade usados para autenticação, autorização e governança: usuários, grupos, dispositivos, aplicações, service principals, roles, políticas e logs. Pense no tenant como o “plano de controle” de identidade do Microsoft 365, Azure e aplicações integradas.

Como isso aparece na prova: a SC-300 costuma perguntar qual configuração fica em tenant settings, user settings, group settings, device settings, company branding, custom domains ou role assignments. A pegadinha é confundir configuração de identidade com configuração de recurso Azure.

1.2 Conceitos base: usuário, grupo, dispositivo e aplicação

Usuário representa uma identidade humana. Pode ser cloud-only, sincronizado do AD DS ou externo. Grupo organiza acesso e licenciamento. Pode ser Security group, Microsoft 365 group, assigned ou dynamic. Dispositivo é um objeto usado em SSO, Conditional Access, compliance e device trust. Aplicação pode aparecer como App Registration, Enterprise Application ou service principal, dependendo do contexto.

1.3 Users, groups and license management

Para criar usuários com qualidade, pense em JML: Joiner, Mover e Leaver. O usuário deve ter UPN correto, display name, usage location, department, job title, country, manager e grupos iniciais. Atributos corretos alimentam dynamic groups, access reviews, licença por grupo e relatórios.

Exemplo de lab: crie Ana Martins como Claim Handler, Bruno Costa como Claim Manager, Carla Ribeiro como Service Desk Analyst e Diego Silva como Developer. O nome é da pessoa; o cargo fica no atributo Job title. Evite nomes como “Ana ClaimHandler”.

Group-based licensing aplica licenças por membership. Isso escala melhor que atribuição manual. A pegadinha é Usage location: se não estiver definido, a licença pode falhar.

Regra dinâmica exemplo:
(user.country -eq "PT") and (user.department -eq "Claims") and (user.accountEnabled -eq true)

1.4 Administrative Units e custom security attributes

Administrative Units reduzem escopo administrativo. Em vez de dar User Administrator no tenant inteiro, você cria uma AU como AU-Portugal e atribui Helpdesk Administrator apenas naquele escopo. Isso é um exemplo direto de least privilege.

Custom security attributes são atributos chave-valor definidos pela organização, úteis para ABAC e segmentação. Exemplo: Project = ClaimsPortal, DataClass = Confidential. Eles ajudam quando grupos não são flexíveis o suficiente para representar contexto de negócio.

Um ponto importante: device registration não é a mesma coisa que Intune enrollment. O device object pode existir no Entra sem estar gerido pelo Intune. Para Conditional Access com compliant device, você precisa que o dispositivo esteja gerenciado e avaliado por uma solução de compliance, normalmente Intune.

1.6 RBAC, permissões e least privilege no Entra

RBAC significa Role-Based Access Control. A pergunta central é: “quem pode fazer o quê, onde e por quanto tempo?”. No Microsoft Entra, uma atribuição de role combina: principal (usuário/grupo/service principal), role (permissão), scope (tenant, AU ou objeto) e, quando usado com PIM, duração/condições.

2.1 Microsoft Entra Connect Sync vs Cloud Sync

Connect Sync é a solução clássica instalada on-premises, com sync engine, connectors, metaverse, regras de sincronização e opções avançadas como device writeback e cenários híbridos complexos. Cloud Sync usa agentes leves e configuração na nuvem, sendo mais simples para múltiplas florestas e ambientes de fusão/aquisição.

2.2 PHS, PTA e Federation/AD FS

Usuário
→
Microsoft Entra sign-in
→
PHS: valida na nuvem
PTA: agente valida no AD
Federation: redireciona ao IdP/AD FS

Password Hash Synchronization sincroniza um hash derivado da senha. Não envia senha em texto claro. É simples, resiliente e permite autenticação cloud mesmo se o ambiente on-prem estiver indisponível.

Pass-through Authentication mantém validação da senha no AD DS local por meio de agentes. É útil quando a empresa precisa aplicar algumas regras on-prem no momento do login, mas depende da disponibilidade dos agentes.

Federation/AD FS redireciona o usuário para um serviço federado. Ainda aparece em ambientes legados ou requisitos específicos, mas para muitos cenários Microsoft recomenda reduzir complexidade migrando para cloud authentication.

Seamless SSO não é método de autenticação isolado. Ele complementa PHS ou PTA para reduzir prompts em dispositivos corporativos na rede, usando Kerberos e a conta AZUREADSSOACC.

2.3 Kerberos em linguagem prática

Kerberos é um protocolo de autenticação baseado em tickets. Em vez de mandar a senha para cada serviço, o usuário obtém tickets de um Key Distribution Center, no AD DS normalmente o Domain Controller. O fluxo simplificado é: o usuário autentica, recebe um TGT, pede um service ticket para um serviço específico e apresenta esse ticket ao serviço.

2.4 B2B, B2C, Direct Connect e Cross-tenant synchronization

ModeloO que resolveCria guest?Exemplo
B2B CollaborationConvidar usuários externos para apps/SharePoint/TeamsSimConsultor externo acessa Claims Portal
B2B Direct ConnectColaboração entre tenants com trust diretoNãoTeams Shared Channels
Azure AD B2C / External IDIdentidade de consumidores/clientesNão é o mesmo tenant corporativoPortal de clientes
Cross-tenant syncProvisionar usuários entre tenantsCria objeto no targetM&A ou multi-tenant organization

Cross-tenant synchronization automatiza a criação/atualização/remoção de usuários de um tenant origem para um tenant destino. Ele não “sincroniza senha” nem junta tenants; ele provisiona objetos para colaboração e acesso. É muito usado quando empresas têm mais de um tenant e precisam de experiência mais integrada.

2.5 External identities configuration

External collaboration settings definem quem pode convidar guests, se guests podem convidar outros guests, nível de visibilidade de guests e allow/deny list de domínios. Cross-tenant access settings controlam inbound/outbound por tenant parceiro e podem confiar em claims de MFA, compliant device e hybrid joined do tenant parceiro.

Para parceiros que não usam Entra ID, você pode configurar identity providers externos com SAML/WS-Fed, incluindo metadados, endpoints e certificado.

A política de métodos de autenticação centraliza quais métodos são permitidos, para quais usuários/grupos, e com quais configurações. Ela é mais moderna que configurações legadas separadas de MFA e SSPR.

3.2 MFA policies e registration

MFA pode ser habilitado por Security Defaults, per-user MFA ou Conditional Access. Para ambiente corporativo maduro, Conditional Access é preferível porque permite granularidade: exigir MFA só para apps, roles, riscos, localizações ou tipos de usuário específicos.

Registration campaign ajuda a migrar usuários para Microsoft Authenticator. Combined registration permite registrar MFA e SSPR na mesma experiência.

3.3 Passwordless: Windows Hello for Business e FIDO2

Windows Hello for Business usa chave assimétrica protegida por TPM e liberada por biometria ou PIN. O PIN não é a senha; ele libera uma chave no device. FIDO2/passkey usa criptografia de chave pública e valida o origin do site, reduzindo phishing.

Temporary Access Pass é ideal para bootstrap: o usuário usa TAP para registrar passkey/WHfB/Authenticator e depois o TAP expira.

3.4 SSPR e password writeback

SSPR reduz tickets de password reset. Em ambiente híbrido, password writeback escreve a nova senha de volta no AD DS. Configure métodos mínimos, número de métodos necessários, registration e notificações.

Na prova, atenção a dois pontos: SSPR não substitui MFA; e password writeback depende de configuração híbrida correta.

3.5 Password Protection

Microsoft Entra Password Protection bloqueia senhas fracas comuns e termos customizados. Para AD DS on-prem, são instalados DC agents e proxy agents. Use modo Audit para medir impacto antes de Enforce.

4.1 Conditional Access fundamentals

Conditional Access segue o modelo: IF usuário/app/condição, THEN controle de acesso. Ele não autentica o usuário sozinho; ele decide se a autenticação precisa de controles adicionais.

Assignments incluem users/groups/roles, target resources, network location, device platform, client app, sign-in risk e device filters. Grant controls incluem block, MFA, authentication strength, compliant device, hybrid joined, approved app e terms of use.

4.2 Policy creation: users, locations, devices and apps

Boas políticas começam com naming convention e rollout em report-only. Exemplo: CA001-Block-LegacyAuth-AllUsers, CA010-Require-MFA-AllUsers, CA020-Admins-PhishingResistant-CompliantDevice.

Break-glass: mantenha pelo menos duas contas de emergência excluídas de CA, altamente monitoradas, com senha forte guardada offline. Elas existem para recuperar o tenant em caso de erro de política.

Device filters permitem aplicar política com base em propriedades do dispositivo. Named locations marcam IPs trusted ou países bloqueados/permitidos.

4.3 Authentication strength, context e protected actions

Authentication strength define quais métodos satisfazem MFA. Para admins, prefira phishing-resistant MFA. Authentication context protege ações ou dados sensíveis dentro de uma aplicação. Protected actions exigem autenticação extra para operações administrativas críticas, como alterar policies sensíveis.

4.4 Microsoft Entra ID Protection e risk policies

ID Protection avalia sign-in risk e user risk. Sign-in risk se refere à tentativa atual: anonymous IP, impossible travel, unfamiliar properties, password spray etc. User risk indica que a identidade como um todo pode estar comprometida, como leaked credentials.

RiscoExemploResposta comum
Sign-in risk mediumLogin suspeitoRequire MFA
Sign-in risk highForte indício maliciosoBlock ou MFA forte
User risk highCredencial vazadaRequire password change

4.5 CAE, token protection e sessão

Continuous Access Evaluation permite revogação quase em tempo real quando há mudança crítica, como usuário desabilitado, senha alterada ou risco elevado. Token protection vincula token ao dispositivo, reduzindo replay de token roubado. Sign-in frequency controla quando reautenticar, mas não substitui CAE.

SSO reduz prompts e centraliza controle no Entra. Para SaaS, os métodos mais comuns são SAML e OpenID Connect/OAuth 2.0. SAML envia assertions com claims; OIDC usa ID tokens e access tokens. Password-based SSO é fallback para apps antigas e deve ser evitado quando possível.

5.2 App registration vs Enterprise Application

App Registration define a aplicação: client ID, redirect URIs, API permissions, secrets/certificates, app roles e token configuration. Enterprise Application é a instância dessa app no tenant, onde você configura SSO, users/groups assignment, Conditional Access e provisioning.

Admin consent workflow permite que usuários solicitem aprovação para apps que exigem permissões elevadas. A prova costuma testar consentimento delegado vs application permission.

5.3 Application Proxy

Microsoft Entra Application Proxy publica aplicações web on-premises sem abrir inbound firewall para a internet. O Private Network Connector faz conexão outbound com o serviço da Microsoft, e o usuário acessa uma URL externa protegida por Entra ID.

Pre-authentication faz o usuário autenticar no Entra antes de chegar à app. Passthrough publica sem pre-auth do Entra e deve ser usado com cuidado. Para apps Windows/IWA, use Kerberos Constrained Delegation para SSO.

5.4 RBAC: Entra roles, Azure RBAC e PIM

Não confunda Microsoft Entra roles com Azure RBAC. Entra roles administram diretório: usuários, grupos, apps, CA, roles. Azure RBAC administra recursos Azure: subscription, resource group, Key Vault, VM, Storage.

5.5 Workload identities, Managed Identity e Key Vault

Managed Identity é uma identidade gerenciada pelo Azure para recursos como VM, Function App, Logic App e Automation Account. Ela evita client secrets hardcoded. Para acessar o Key Vault, habilite a identidade no recurso e dê permissão com Azure RBAC, como Key Vault Secrets User.

# Exemplo conceitual
1. Habilitar system-assigned managed identity na Function App
2. No Key Vault, atribuir role: Key Vault Secrets User
3. No código, solicitar token via DefaultAzureCredential
4. Ler o secret sem armazenar password/client secret

Workload Identity Federation permite que GitHub Actions, Azure DevOps ou Kubernetes autentiquem no Entra sem segredo, trocando um token externo confiável por token Entra.

Privileged Identity Management reduz privilégio permanente. Em vez de um admin ficar Global Administrator 24/7, ele fica eligible e ativa a role quando necessário. A ativação pode exigir MFA, justificativa, aprovação, ticket number e duração limitada.

6.2 Just-in-Time access and approval workflows

JIT significa acesso apenas quando necessário. Configure duração curta para roles críticas, approval para roles sensíveis e alertas para ativações suspeitas. Para tarefas rotineiras, crie roles específicas em vez de usar Global Admin.

Na prova, se o requisito menciona “reduzir privilégio permanente” ou “elevação temporária”, a resposta tende a envolver PIM.

Access Reviews validam se usuários ainda precisam de acesso. Podem revisar membros de grupos, acesso a Enterprise Apps, guests, access packages e roles privilegiadas. Reviews podem usar recomendações, como remover usuários inativos.

6.4 Entitlement Management e Access Packages

Entitlement Management organiza acesso em catalogs, access packages e policies. Um access package pode incluir grupos, apps e sites SharePoint. A policy define quem pode solicitar, quem aprova, duração, expiração e revisão.

Connected organizations representam parceiros externos que podem solicitar access packages. Isso é muito útil para governança B2B.

6.5 Lifecycle Workflows

Lifecycle Workflows automatizam tarefas de joiner, mover e leaver com base em atributos como employeeHireDate ou employeeLeaveDateTime. Exemplos: enviar e-mail de boas-vindas, adicionar usuário a grupos, remover licenças, desabilitar conta e acionar access reviews.
