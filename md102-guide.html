Domínio 1

Implantar o cliente Windows

25–30% do exame

Este domínio é sobre como o Windows chega ao dispositivo do utilizador. Cobre desde a escolha da edição correta do Windows 11, passando pelos métodos de ativação, até às ferramentas modernas de provisionamento como o Windows Autopilot (que terá uma secção dedicada mais à frente).

1.1 · Edições do Windows 11 e Licenciamento

O primeiro passo de qualquer implantação é escolher a edição certa do Windows. Cada edição tem capacidades e públicos diferentes:

Edição Para quem é Pode ingressar em Entra ID? BitLocker
Home Utilizadores domésticos. Não suportado em ambiente corporativo. Não Não
Pro Pequenos negócios e profissionais Sim Sim
Enterprise Grandes empresas (E3/E5). Tem AppLocker, Credential Guard, DirectAccess. Sim Sim
Education Equivalente ao Enterprise, mas para instituições de ensino Sim Sim
Enterprise LTSC Dispositivos especializados (caixas registadoras, equipamentos médicos). Sem Microsoft Store, sem updates de funcionalidades frequentes. Sim Sim
💡 Para a prova

Se vires um cenário com Windows 11 Home, a resposta correta envolve quase sempre fazer upgrade para Pro antes de qualquer ingresso em Entra ID ou inscrição no Intune. Home não suporta gestão empresarial.

Métodos de Ativação do Windows

Existem quatro grandes métodos de ativação que precisas de conhecer:

  • MAK (Multiple Activation Key): Uma chave única que ativa um número limitado de dispositivos. Cada ativação consome uma "unidade" da chave. Usado em ambientes pequenos ou desconectados.
  • KMS (Key Management Service): Servidor on-premises que ativa clientes na rede interna. Os clientes precisam contactar o servidor KMS pelo menos uma vez a cada 180 dias para manterem a ativação.
  • Active Directory-Based Activation (ADBA): A ativação é feita automaticamente quando o dispositivo se junta ao domínio AD. Não precisa de um servidor KMS dedicado.
  • Subscription Activation: O dispositivo "sobe" automaticamente de edição (Pro → Enterprise) quando o utilizador faz login com uma conta Entra ID com licença Windows 11 Enterprise E3/E5. É a forma moderna e a mais cobrada na prova.
Conceito-chave

Subscription Activation é o método preferido em ambientes cloud-first. O dispositivo vem com Windows 11 Pro de fábrica, e ao entrar com a conta Entra ID licenciada, é automaticamente promovido a Enterprise sem necessidade de imagem ou chave de produto. Quando o utilizador sai ou perde a licença, o dispositivo volta para Pro.

1.2 · Métodos de Implantação do Windows

Existem várias formas de instalar o Windows num dispositivo. Vou apresentá-las da mais moderna (cloud-first) à mais tradicional (on-premises):

Windows Autopilot (cloud-first, moderno)

Permite que dispositivos novos saiam direto do fornecedor para o utilizador final, e que tudo seja configurado automaticamente quando o utilizador liga o dispositivo pela primeira vez. Tem uma secção dedicada mais à frente neste guia.

Provisioning Packages (Windows Configuration Designer)

Ficheiros .ppkg criados na ferramenta Windows Configuration Designer. Podem ser aplicados por:

  • Pen USB
  • Email
  • Partilha de rede
  • Diretamente durante o OOBE (Out-of-Box Experience)

Útil para configurar Wi-Fi, certificados, políticas e até para ingressar dispositivos em Entra ID em massa, sem precisar de Intune. Bom para cenários offline ou redes com limitação.

Configuration Manager (anteriormente SCCM)

Solução on-premises da Microsoft para implantação em larga escala. Suporta:

  • Task sequences (sequências de tarefas) para imaginar dispositivos
  • OSD (Operating System Deployment) com PXE boot
  • Integração com Intune via co-management

MDT (Microsoft Deployment Toolkit) e WDS (Windows Deployment Services)

Ferramentas tradicionais para criar e implantar imagens via rede usando PXE boot. Ainda existem mas são consideradas legacy. A Microsoft recomenda fortemente migrar para Autopilot.

1.3 · Caminhos de Upgrade e Feature Updates

Existem três tipos principais de mover-se entre versões do Windows:

  • In-place upgrade: Atualizar mantendo apps, ficheiros e definições. Suportado de Windows 10 → Windows 11 (se o hardware cumprir os requisitos: TPM 2.0, Secure Boot, CPU compatível).
  • Wipe-and-load: Apaga tudo e instala uma imagem nova. Usado em refresh de hardware antigo.
  • Side-by-side migration: Migração entre dois dispositivos (substituição de hardware).
⚠️ Requisitos do Windows 11

Tens de conhecer estes requisitos para a prova. Um dispositivo só pode ter Windows 11 se cumprir:

  • TPM 2.0 (Trusted Platform Module)
  • Secure Boot ativado (UEFI)
  • CPU compatível (lista da Microsoft)
  • Mínimo 4 GB RAM e 64 GB de armazenamento
  • DirectX 12 / WDDM 2.0

Se um dispositivo não cumpre, fica no Windows 10 ou tem de ser substituído. Não há "workaround" oficial.

Feature Updates vs Quality Updates

Tipo O que é Frequência Onde se configura
Feature Update Atualização de versão (ex: 23H2 → 24H2). Adiciona novas funcionalidades. Anual Intune → Feature updates for Windows 10 and later
Quality Update Correções de segurança e bugs. O famoso "Patch Tuesday". Mensal Intune → Quality updates ou Update rings
Driver Update Atualizações de drivers via Windows Update for Business Quando disponível Intune → Driver updates

1.4 · Windows 365 e Azure Virtual Desktop

Estes são serviços de Cloud PC – ou seja, em vez do Windows correr no portátil físico, corre numa máquina virtual na nuvem que o utilizador acede remotamente.

Característica Windows 365 Azure Virtual Desktop (AVD)
Modelo Subscrição fixa por utilizador (SaaS) Pay-as-you-go (consumo Azure)
Multi-sessão Não – 1 PC por utilizador Sim – vários utilizadores num host
Gestão Microsoft Intune (mais simples) Portal do Azure (mais complexo, mais flexível)
Caso de uso Trabalhador remoto que precisa de "um PC" na nuvem VDI tradicional, cenários com customização pesada
Como decidir

Se a questão fala em "experiência semelhante a um PC pessoal", "simples de gerir" ou "licenciamento previsível por utilizador" → Windows 365. Se fala em "vários utilizadores partilhando recursos", "otimização de custos" ou "multi-sessão" → Azure Virtual Desktop.

Questão 1

Um utilizador comprou um portátil novo com Windows 11 Home. Tu queres ingressá-lo em Microsoft Entra ID e inscrevê-lo no Intune. Qual é o primeiro passo?

  1. A. Ingressar o dispositivo diretamente em Entra ID via Settings.
  2. B. Fazer upgrade da edição para Windows 11 Pro (ou superior) antes do ingresso.
  3. C. Usar um provisioning package para forçar o ingresso.
  4. D. Inscrever o dispositivo via Company Portal.
✓ Resposta: B Windows 11 Home não suporta ingresso em Entra ID nem inscrição no Intune. É necessário fazer upgrade para Pro, Enterprise ou Education primeiro. Esta é uma armadilha clássica da prova: aparece um cenário corporativo com edição Home e a resposta é sempre "upgrade primeiro".
Questão 2

A tua empresa quer que os dispositivos sejam automaticamente promovidos de Windows 11 Pro para Enterprise quando os utilizadores fazem login com a conta Entra ID. Qual método de ativação devo usar?

  1. A. KMS
  2. B. MAK
  3. C. Subscription Activation
  4. D. Active Directory-Based Activation
✓ Resposta: C Subscription Activation é o método cloud-first que aproveita as licenças Windows 11 Enterprise E3/E5 atribuídas ao utilizador via Entra ID. Quando o utilizador faz login, a edição "sobe" de Pro para Enterprise. KMS e MAK exigem chaves manuais; ADBA exige domínio AD on-premises.
Questão 3

Tens uma frota de 50 dispositivos que precisam ser configurados para uma rede Wi-Fi específica, com certificado, sem acesso à internet pública durante o setup. Qual é a melhor abordagem?

  1. A. Windows Autopilot self-deploying mode
  2. B. Provisioning package via Windows Configuration Designer
  3. C. Configuration Manager task sequence
  4. D. Microsoft Deployment Toolkit (MDT)
✓ Resposta: B Provisioning packages (ficheiros .ppkg) são ideais quando precisas de configurar dispositivos offline ou com conectividade limitada. Podem ser aplicados via USB diretamente no OOBE e configurar Wi-Fi e certificados antes do dispositivo ter acesso à internet. Autopilot precisa sempre de internet para contactar o serviço Microsoft.
Questão 4

Um dispositivo com Windows 10 não cumpre os requisitos de hardware do Windows 11 (sem TPM 2.0). Qual é a recomendação oficial da Microsoft?

  1. A. Forçar o upgrade desativando a verificação de TPM via registry.
  2. B. Instalar Windows 11 via ISO ignorando os requisitos.
  3. C. Manter no Windows 10 ou substituir o hardware – não há caminho suportado de upgrade.
  4. D. Usar Subscription Activation para forçar o Windows 11.
✓ Resposta: C A Microsoft não suporta Windows 11 em hardware que não cumpre TPM 2.0 + Secure Boot + CPU compatível. Mesmo que existam workarounds, a resposta da prova é sempre "manter no Windows 10 (com ESU se necessário) ou substituir hardware". Subscription Activation muda a edição, não o sistema operativo.
Questão 5

A tua organização quer disponibilizar máquinas virtuais Windows na nuvem com licenciamento previsível por utilizador, gestão via Intune e cada utilizador com o seu próprio "PC". Que serviço escolhes?

  1. A. Azure Virtual Desktop multi-sessão
  2. B. Windows 365 Cloud PC
  3. C. Hyper-V on-premises
  4. D. Azure VM Standard
✓ Resposta: B Windows 365 é a opção SaaS, 1 PC por utilizador, com subscrição mensal fixa e gestão via Intune. AVD seria para multi-sessão (vários utilizadores partilham um host) e tem licenciamento pay-as-you-go. As palavras-chave aqui são "previsível", "seu próprio PC" e "Intune".
Questão 6

Qual é a diferença entre Feature Update e Quality Update no Windows?

  1. A. Feature Update é mensal, Quality Update é anual.
  2. B. Feature Update muda a versão do Windows (ex: 23H2 → 24H2); Quality Update traz correções mensais.
  3. C. Ambos são opcionais e não obrigatórios.
  4. D. Feature Update é apenas para drivers; Quality Update é para apps.
✓ Resposta: B Feature Updates são as atualizações anuais que mudam a versão (23H2, 24H2, etc.). Quality Updates são as mensais de segurança e bug fixes (Patch Tuesday). No Intune são configurados em políticas separadas: "Feature updates for Windows 10 and later" vs "Update rings" / "Quality updates".
Questão 7

Estás a planear migrar 200 dispositivos de Windows 10 para Windows 11. Os utilizadores não podem perder aplicações nem ficheiros. Qual método usas?

  1. A. In-place upgrade
  2. B. Wipe-and-load
  3. C. Side-by-side migration
  4. D. Autopilot Reset
✓ Resposta: A In-place upgrade mantém apps, ficheiros e definições. Wipe-and-load apaga tudo. Side-by-side é entre dois dispositivos físicos diferentes. Autopilot Reset traz o dispositivo de volta para o estado inicial sem reimaginar (é usado para repurpose, não para upgrade de versão).
Questão 8

Qual edição do Windows 11 deves escolher para um quiosque numa loja, onde o dispositivo nunca deve receber feature updates frequentes e não precisa da Microsoft Store?

  1. A. Windows 11 Pro
  2. B. Windows 11 Enterprise
  3. C. Windows 11 Enterprise LTSC
  4. D. Windows 11 Education
✓ Resposta: C LTSC (Long-Term Servicing Channel) é desenhado para dispositivos especializados que precisam de estabilidade máxima: quiosques, equipamentos médicos, ATMs, máquinas industriais. Não recebe feature updates frequentes e não tem Microsoft Store. Palavras-chave: "quiosque", "industrial", "sem updates frequentes".
Questão 9

Numa empresa com Active Directory on-premises, sem servidor KMS dedicado, qual é a forma mais simples de ativar centenas de Windows 11 Pro?

  1. A. Distribuir uma chave MAK para cada técnico instalar manualmente.
  2. B. Active Directory-Based Activation (ADBA)
  3. C. Subscription Activation
  4. D. Active a cada dispositivo via OEM
✓ Resposta: B ADBA permite ativar dispositivos automaticamente quando ingressam no domínio AD, sem precisar de um servidor KMS dedicado. Subscription Activation funciona para Pro → Enterprise (não para ativar o Pro propriamente dito) e exige Entra ID + licenças E3/E5. MAK exigiria distribuição manual de chaves.
Questão 10

Verdadeiro ou falso: Configuration Manager (anteriormente SCCM) é uma solução cloud-first da Microsoft.

  1. A. Verdadeiro
  2. B. Falso – é uma solução on-premises, embora possa ser integrada com Intune via co-management.
✓ Resposta: B Configuration Manager é uma solução on-premises. A Microsoft criou o co-management exatamente para permitir migração gradual: o dispositivo é gerido em simultâneo pelo ConfigMgr (workloads on-prem) e pelo Intune (workloads cloud). Cada workload pode ser movido individualmente para o Intune.
Domínio 2

Gerir identidade e conformidade

15–20% do exame

Este domínio cobre como o dispositivo "se apresenta" ao Microsoft Entra ID e como garantes que ele cumpre regras de segurança antes de aceder a recursos corporativos. Vais ver aqui a parte mais confusa para muita gente: os tipos de ingresso (join types).

2.1 · Os 3 estados de dispositivo no Entra ID

Esta é provavelmente a área mais cobrada da prova MD-102 e onde a maior parte das pessoas se baralha. Vou explicar com calma.

Quando um dispositivo "aparece" no Entra ID, ele pode estar num de três estados:

  1. Microsoft Entra Joined (Entra Joined)
  2. Microsoft Entra Hybrid Joined (Hybrid Joined)
  3. Microsoft Entra Registered (Registered, também chamado Workplace Join)
A ideia central

A diferença entre eles é quem é o "dono" da identidade do dispositivo e que tipo de utilizador faz login nele. Vou explicar cada um separadamente, com analogia, e depois fazer uma tabela comparativa.

🔷 Entra Joined

O dispositivo é "cidadão" do Entra ID. Não tem ligação com Active Directory on-premises. O utilizador faz login no Windows com a sua conta Entra ID corporativa (nome@empresa.com).

  • Quem se autentica: a conta de trabalho Entra ID
  • Dispositivo é propriedade de: a organização (corporate-owned)
  • Sistema operativo: Windows 10/11 Pro, Enterprise, Education (Home NÃO suporta)
  • Recebe Primary Refresh Token (PRT) no login → SSO automático para Microsoft 365 e apps SaaS
  • Gestão: Microsoft Intune (MDM completo)
  • Casos de uso: ambiente cloud-first, empresas modernas, startups, qualquer organização sem dependência forte de apps legacy on-premises

🔶 Entra Hybrid Joined

O dispositivo está "em duas casas" ao mesmo tempo: ingressado no Active Directory on-premises e também registado no Entra ID. É o estado da "ponte" para empresas que ainda têm AD on-prem mas querem benefícios do Entra (SSO cloud, Conditional Access, Intune).

  • Quem se autentica: a conta AD (que é sincronizada com Entra ID via Entra Connect ou Entra Cloud Sync)
  • Dispositivo é propriedade de: a organização
  • Sistema operativo: apenas Windows (Windows 10/11 Pro, Enterprise, Education)
  • Precisa de linha de visão para Domain Controller on-premises (problema para remote workers!)
  • Suporta Kerberos/NTLM para apps legacy on-premises (file shares, apps internas)
  • Group Policy (GPO) continua a funcionar
  • Casos de uso: empresas em transição, apps legacy que dependem de AD, file shares Kerberos
⚠️ Recomendação da Microsoft

A Microsoft já não recomenda implantar novos dispositivos como Hybrid Joined. Para novos dispositivos, a recomendação oficial é Entra Joined (cloud-native). Hybrid Joined existe principalmente para suportar quem ainda tem AD legacy. Isto aparece na prova: se a questão pergunta "qual o estado recomendado para NOVOS dispositivos", a resposta é Entra Joined.

🟡 Entra Registered (Workplace Join)

O dispositivo é pessoal – pertence ao utilizador, não à organização. O utilizador apenas "regista" o dispositivo para conseguir aceder a recursos da empresa (email, Teams, etc.) sem perder o controlo do seu próprio dispositivo.

  • Quem se autentica: conta pessoal (Microsoft account local) + uma conta de trabalho adicional para apps específicas
  • Dispositivo é propriedade de: o utilizador (BYOD — Bring Your Own Device)
  • Sistema operativo: ⭐ Windows 10/11, macOS, iOS, iPadOS, Android, Linux – é o ÚNICO estado que suporta TUDO
  • Gestão: normalmente MAM (Mobile Application Management), não MDM completo. A organização controla apenas as apps corporativas, não o dispositivo todo.
  • SSO: mais limitado – funciona para apps cloud específicas, não é "transparente" como nos outros estados
  • Casos de uso: BYOD, telefones pessoais a aceder a email corporativo, contractors

Identifica pelas palavras-chave do cenário:

2.2 · Autenticação no dispositivo

Windows Hello for Business (WHfB)

Substitui passwords por autenticação biométrica (impressão digital, reconhecimento facial) ou PIN. Importante: o PIN do WHfB é diferente de uma password — está ligado ao TPM do dispositivo e nunca sai do hardware.

  • Cloud trust model (recomendado): usa Entra ID Kerberos. Não precisa de PKI on-premises.
  • Hybrid certificate trust: usa certificados emitidos por uma CA on-premises.
  • Hybrid key trust: usa chaves armazenadas no TPM, requer configuração de Kerberos.

FIDO2 / Passkeys

Chaves de segurança físicas (USB, NFC) ou passkeys armazenadas em dispositivos. Permitem login passwordless completamente.

Multi-Factor Authentication (MFA)

Combina algo que sabes (password) + algo que tens (telemóvel, token) + algo que és (biometria). Configurado em Conditional Access ou via Security Defaults.

2.3 · Compliance Policies (Políticas de Conformidade)

Uma Compliance Policy no Intune define quais regras um dispositivo deve cumprir para ser considerado "compliant". Exemplos:

  • Versão mínima do sistema operativo
  • BitLocker ativado
  • Secure Boot ativado
  • Antivirus em dia
  • Sem jailbreak/root
  • Password com complexidade mínima
  • Microsoft Defender for Endpoint com risk score baixo
Estado do dispositivo

Cada dispositivo é avaliado periodicamente e marcado como Compliant ✅ ou Non-compliant ❌. Este estado é enviado ao Entra ID e pode ser usado em políticas de Conditional Access.

Actions for noncompliance

Quando um dispositivo se torna non-compliant, podes configurar ações graduais:

  1. Mark device noncompliant imediatamente, ou após X dias de tolerância
  2. Send email to end user a alertar
  3. Send push notification ao utilizador
  4. Remotely lock the device
  5. Retire the noncompliant device (remoção completa)

2.4 · Conditional Access (CA)

É o "porteiro" do Entra ID. Uma política de Conditional Access tem dois lados:

Condições (Quando?) Controlos (O quê?)
Utilizadores ou grupos Bloquear acesso (Block)
Apps ou recursos Exigir MFA
Localização (IP, país) Exigir dispositivo compliant
Plataforma do dispositivo Exigir dispositivo Hybrid Joined
Estado do dispositivo Exigir app aprovada (ex: Outlook)
Risco do sign-in / utilizador Exigir mudança de password
💡 AND vs OR nos controlos

Os controlos de "Grant" podem ser combinados:

  • Require all of the selected controls (AND): "Exigir MFA E dispositivo compliant" — muito mais restritivo
  • Require one of the selected controls (OR): "Exigir MFA OU dispositivo compliant" — menos restritivo

A prova adora testar isto. Lê com atenção qual operador está a ser usado.

Report-only mode

Antes de "ligar" uma política em produção, podes pô-la em Report-only. A política é avaliada para cada sign-in mas não é aplicada – só regista no log o que teria acontecido. Excelente para testar sem partir nada.

2.5 · Intune RBAC e Scope Tags

Em organizações grandes, queres separar quem pode gerir o quê. O Intune tem três conceitos:

  • Roles: "School Administrator", "Help Desk Operator", "Policy and Profile Manager", etc.
  • Scope Tags: etiquetas que limitam quais recursos um admin vê. Ex: scope tag "Lisbon" só vê dispositivos do escritório de Lisboa.
  • Assignments: a quem o role/scope é atribuído.

🎯 Mini Quiz · Domínio 2

10 questões com foco especial nas diferenças entre os tipos de join. Clica numa opção para responder.

Questão 1

Um utilizador da tua empresa quer aceder ao email corporativo no seu iPhone pessoal. Ele não quer que a empresa controle todo o telefone, apenas o email. Qual estado de dispositivo é apropriado?

  1. A. Entra Joined
  2. B. Hybrid Joined
  3. C. Entra Registered
  4. D. Nenhum – iPhone não suporta Entra
✓ Resposta: C Entra Registered é o ÚNICO estado que suporta iOS (e macOS, Android, Linux). É feito para cenários BYOD: o dispositivo é do utilizador, e a empresa apenas controla as apps corporativas via MAM (Mobile Application Management). Entra Joined e Hybrid Joined exigem Windows.
Questão 2

A tua organização tem uma aplicação interna que exige autenticação Kerberos contra Active Directory. Qual estado de dispositivo deves usar para permitir que utilizadores remotos a acedam, mantendo benefícios do Entra ID?

  1. A. Entra Joined
  2. B. Hybrid Joined
  3. C. Entra Registered
  4. D. Workplace Joined
✓ Resposta: B Hybrid Joined é a única opção que suporta Kerberos/NTLM nativo contra AD on-premises e também tem benefícios do Entra ID (Conditional Access, Intune, SSO cloud). Entra Joined puro não tem Kerberos nativo para apps on-prem (exigiria Azure AD Kerberos ou outro mecanismo).
Questão 3

Crias uma política de Conditional Access que exige "MFA AND device compliant". Um utilizador faz MFA mas o dispositivo está non-compliant. O que acontece?

  1. A. Acesso permitido – MFA é suficiente
  2. B. Acesso negado – ambos os controlos têm de ser cumpridos
  3. C. Acesso permitido com aviso
  4. D. Acesso permitido em modo somente leitura
✓ Resposta: B Quando usas "Require all of the selected controls" (AND), todos os controlos têm de passar. Se o dispositivo está non-compliant, a política bloqueia mesmo que o MFA tenha passado. Se quisesses que MFA bastasse, terias de usar "OR".
Questão 4

Queres testar uma nova política de Conditional Access sem afetar utilizadores reais. Qual é a melhor abordagem?

  1. A. Aplicar a política apenas ao teu utilizador admin
  2. B. Desativar a política depois de criada
  3. C. Configurar a política em modo Report-only
  4. D. Aplicar a política a um grupo vazio
✓ Resposta: C Report-only mode é desenhado exatamente para isto: a política é avaliada mas não enforced. Vês nos logs o que teria acontecido sem afetar ninguém. Quando estiveres confiante, mudas para "On".
Questão 5

Numa Compliance Policy, o que acontece quando configuras "Mark device noncompliant" com tolerância de 5 dias?

  1. A. O dispositivo é marcado non-compliant imediatamente
  2. B. O dispositivo tem 5 dias para voltar a estar compliant antes de ser marcado non-compliant
  3. C. O dispositivo é apagado depois de 5 dias
  4. D. O utilizador é bloqueado por 5 dias
✓ Resposta: B A tolerância (grace period) dá tempo ao utilizador para corrigir o problema (ex: instalar uma atualização) antes do dispositivo ser oficialmente marcado non-compliant. Durante esses 5 dias, o dispositivo é considerado "in grace period".
Questão 6

Um administrador na filial de Lisboa só deve poder gerir dispositivos localizados em Lisboa. Como o configuras no Intune?

  1. A. Criar uma subscrição Intune separada para Lisboa
  2. B. Usar Scope Tags com o valor "Lisbon" e atribuir o role RBAC com esse scope
  3. C. Criar uma Compliance Policy específica para Lisboa
  4. D. Não é possível segmentar admins no Intune
✓ Resposta: B Scope Tags são a forma do Intune segmentar visibilidade de recursos. Atribuis a scope tag "Lisbon" aos dispositivos/políticas de Lisboa, e atribuis o admin a um role RBAC limitado a essa scope tag. Ele só vê o que está marcado com "Lisbon".
Questão 7

A tua empresa está a criar do zero – cloud-first, sem nenhuma infraestrutura on-premises. Qual estado de dispositivo deves implantar para os portáteis Windows novos?

  1. A. Entra Joined
  2. B. Hybrid Joined
  3. C. Entra Registered
  4. D. Domain Joined
✓ Resposta: A Cenário cloud-first → Entra Joined. Não há AD on-premises, logo Hybrid Joined não faz sentido (e exige AD!). Entra Registered seria para BYOD pessoal. A Microsoft recomenda fortemente Entra Joined para novos dispositivos corporativos.
Questão 8

Verdadeiro ou falso: O Windows Hello for Business em modo "cloud trust" exige uma PKI on-premises.

  1. A. Verdadeiro
  2. B. Falso – cloud trust usa Entra ID Kerberos, sem necessidade de PKI on-premises
✓ Resposta: B Cloud trust é o modelo moderno e simples: usa Entra ID Kerberos, sem PKI on-premises. Certificate trust (legacy) é que exige CA on-premises a emitir certificados. Key trust usa chaves no TPM mas precisa de Kerberos AD configurado.
Questão 9

Um dispositivo Entra Joined recebe automaticamente algo no login que permite SSO seamless a Microsoft 365 e apps SaaS. O que é?

  1. A. Certificado SSL
  2. B. Primary Refresh Token (PRT)
  3. C. Kerberos ticket
  4. D. SAML token
✓ Resposta: B O Primary Refresh Token (PRT) é emitido pelo Entra ID no momento do login do Windows e fica armazenado no dispositivo. Permite obter tokens de acesso para qualquer app Entra ID-integrada sem o utilizador precisar autenticar-se novamente. É a magia por trás do SSO em dispositivos Entra Joined e Hybrid Joined.
Questão 10

Quais sistemas operativos podem estar em estado Entra Registered? (escolhe a resposta mais completa)

  1. A. Apenas Windows 10/11
  2. B. Windows e macOS
  3. C. Windows, macOS, iOS, iPadOS, Android, Linux
  4. D. Apenas iOS e Android
✓ Resposta: C Entra Registered é o estado mais "universal" – suporta praticamente todos os sistemas operativos modernos. Foi desenhado para BYOD, então faz sentido suportar qualquer dispositivo que o utilizador possa ter. Entra Joined e Hybrid Joined são exclusivos do Windows (em ambiente MD-102).
Domínio 3

Gerir, manter e proteger dispositivos

40–45% do exame · o mais pesado

Este é o maior domínio da prova e onde tens de te focar mais. Cobre tudo o que acontece depois de o dispositivo estar inscrito no Intune: configuração, segurança, updates, monitorização, Defender, Windows 365, BitLocker e — o tema principal — Windows Autopilot.

3.1 · Enrollment (Inscrição) de Dispositivos

O enrollment é o processo pelo qual um dispositivo "entra" no Intune para ser gerido. Cada sistema operativo tem o seu método:

Plataforma Método de enrollment Notas
Windows 10/11 Autopilot · Automatic enrollment via Entra join · Manual via Company Portal · GPO (Hybrid Join) Suporta tudo
macOS Apple Automated Device Enrollment (ADE/DEP) · Company Portal · Direct enrollment Precisa de Apple Business Manager / Apple School Manager
iOS / iPadOS ADE · Apple Configurator · Company Portal · User enrollment (BYOD) Mesma lógica do macOS
Android Enterprise Fully Managed (corporate) · Dedicated (kiosk) · Corporate-Owned Work Profile (COPE) · Work Profile (BYOD) 4 modelos distintos – decora-os
Linux Apenas Ubuntu LTS (22.04+, 20.04 limitado), com Microsoft Intune App Funcionalidade limitada

Os 4 modos do Android Enterprise

  • Fully Managed: dispositivo 100% da empresa, sem espaço pessoal. Controlo total.
  • Dedicated: dispositivo de uso único (kiosk, leitor de códigos, terminal). Sem utilizador associado.
  • Corporate-Owned with Work Profile (COPE): dispositivo da empresa mas o utilizador também o usa pessoalmente. Espaço pessoal e profissional separados.
  • Work Profile (Personally-Owned): BYOD. O dispositivo é do utilizador; apenas o "work profile" é gerido.

3.2 · Configuration Profiles

São políticas que aplicam configurações específicas aos dispositivos. Os tipos mais importantes:

  • Templates: conjuntos pré-construídos para Wi-Fi, VPN, certificados, email, etc.
  • Settings Catalog: ⭐ o método moderno e recomendado. Permite escolher milhares de definições individualmente. Substitui gradualmente os templates.
  • Administrative Templates: equivalente a GPOs ADMX no Intune (políticas Office, Edge, etc.)
  • Custom (OMA-URI): quando precisas configurar algo muito específico que não está nas outras categorias.

Security Baselines

São conjuntos pré-configurados de definições de segurança recomendadas pela Microsoft. Funcionam como um "ponto de partida seguro". Existem baselines para:

  • Windows 10/11
  • Microsoft Defender for Endpoint
  • Microsoft Edge
  • Microsoft 365 Apps

3.3 · Remote Actions (Ações Remotas)

Ações que podes executar num dispositivo gerido sem o tocar fisicamente. Memoriza estas, pois aparecem muito:

Ação O que faz Dados do utilizador
Sync Força o dispositivo a verificar políticas com o Intune Mantém
Restart Reinicia o dispositivo Mantém
Remote Lock Bloqueia o dispositivo Mantém
Reset Password Define uma nova password temporária Mantém
Wipe Apaga tudo, devolve ao estado de fábrica. Pode preservar a conta de utilizador (opcional). Apaga
Retire Remove apenas dados/apps corporativas. Dados pessoais ficam intactos. Ideal para BYOD. Pessoais ficam
Fresh Start (Windows) Remove apps pré-instaladas pelo OEM, mantendo Windows limpo. Pode preservar dados. Opcional
Autopilot Reset Limpa o dispositivo e reaplica o perfil Autopilot. Reutilização rápida. Apaga
💡 Wipe vs Retire

Confunde muita gente. Memoriza assim:

  • Wipe = funcionário deixou a empresa com dispositivo corporativo → apaga tudo
  • Retire = funcionário deixou a empresa com dispositivo pessoal (BYOD) → apaga só o corporativo

3.4 · Microsoft Defender for Endpoint (MDE)

É a plataforma EDR (Endpoint Detection and Response) da Microsoft. Vai muito além do antivírus tradicional. Componentes principais:

  • Next-gen antivirus: proteção em tempo real, cloud-delivered protection
  • EDR (Endpoint Detection and Response): deteta comportamentos suspeitos, não apenas malware conhecido
  • Attack Surface Reduction (ASR) rules: bloqueia técnicas comuns de ataque (macros, scripts, ransomware behaviour)
  • Threat & Vulnerability Management (TVM): identifica vulnerabilidades nos dispositivos
  • Automated Investigation and Remediation (AIR): resposta automática a incidentes

Onboarding de dispositivos no Defender

  • Via Intune: Configurar a "Endpoint detection and response" policy e atribuir aos grupos. É a forma recomendada.
  • Via Group Policy: para Hybrid Joined
  • Via script local: para dispositivos isolados
  • Via Configuration Manager

Conditional Access baseado em risco

O MDE atribui um Machine Risk Score a cada dispositivo (Low, Medium, High). Podes integrar isto numa Compliance Policy para considerar non-compliant um dispositivo com risk score "Medium" ou superior. Isto, combinado com Conditional Access, cria um ciclo de defesa automático.

3.5 · Windows Autopatch e Update Management

Update Rings

Permite definir anéis de implementação graduais para feature e quality updates:

  1. Pilot ring: 5-10 utilizadores. Recebem updates primeiro.
  2. Early adopters: ~20% do parque. Recebem 1 semana depois.
  3. Broad deployment: ~70% do parque. Recebem 2 semanas depois.
  4. Critical (last): dispositivos críticos. Recebem por último.

Windows Autopatch

Serviço gerido pela Microsoft que automatiza completamente a gestão de updates: Windows, Microsoft 365 Apps, Edge e Teams. Define os rings automaticamente e monitoriza o sucesso da implementação. Requer licença Windows 11 Enterprise E3/E5 ou similar.

3.6 · BitLocker e Encryption

Cifra o disco completo, protegendo os dados se o dispositivo for roubado ou perdido.

  • TPM only: o dispositivo desbloqueia automaticamente no boot (sem PIN). Simples mas menos seguro contra ataques físicos.
  • TPM + PIN: o utilizador insere um PIN no boot. Mais seguro.
  • Recovery key: chave de 48 dígitos que pode desbloquear o dispositivo. É guardada automaticamente no Entra ID quando o dispositivo é Entra Joined ou Hybrid Joined.
Onde encontrar a recovery key

Para Entra Joined: Entra admin center → Devices → escolher o dispositivo → "BitLocker keys". Para Hybrid Joined: pode estar guardada no AD on-premises ou no Entra (depende da configuração).

3.7 · Endpoint Analytics

Ferramenta do Intune que dá insights sobre a experiência do utilizador:

  • Startup performance: quanto tempo o Windows demora a arrancar
  • Application reliability: que apps crasham mais
  • Work from anywhere: quantos dispositivos cumprem requisitos para Windows 11, estão cloud-managed, etc.
  • Resource performance: uso de CPU, RAM, disco

Atribui um score de 0 a 100 a cada dispositivo, ajudando a identificar problemas antes do utilizador reclamar.

3.8 · Windows 365 Cloud PC (revisão profunda)

Já vimos Windows 365 no Domínio 1 a um nível alto. Aqui vamos aprofundar a parte de gestão:

Tipos de Cloud PC

  • Windows 365 Enterprise: versão gerida via Intune, integrada com Entra ID. É o foco da prova.
  • Windows 365 Frontline: para shift workers — várias pessoas partilham a mesma licença em horários diferentes.

Provisioning Policies

3.9 · Windows Autopilot · Mergulho Profundo

Esta secção é dedicada porque pediste detalhamento profundo, e porque Autopilot aparece em quase todas as provas MD-102. Vou explicar tudo: o que é, como funciona internamente, todos os modos, o que cada um exige, e no fim tens um lab para montar.

O que é o Windows Autopilot

Imagina o cenário tradicional: um técnico de IT recebe um portátil novo, instala uma imagem do Windows (golden image), configura o domínio, instala apps, configura políticas, e só depois o envia para o utilizador. Demora horas e exige um técnico físico.

O Autopilot vira esta história ao contrário: o portátil vai direto do fornecedor para o utilizador final. Quando o utilizador liga o portátil pela primeira vez, conecta-se à internet e insere as credenciais Entra ID, o Autopilot:

  1. Reconhece o dispositivo pelo seu hardware hash (já pré-registado no serviço)
  2. Aplica um perfil de Autopilot com configurações de OOBE
  3. Junta o dispositivo ao Entra ID
  4. Inscreve o dispositivo automaticamente no Intune
  5. Instala apps obrigatórias, políticas, certificados, BitLocker, etc.
  6. Entrega o dispositivo "pronto a usar" diretamente ao utilizador

Hardware Hash – a "impressão digital" do dispositivo

Cada dispositivo Windows tem uma identificação única gerada a partir do hardware (CPU, motherboard, TPM, MAC address). Este "hash" é o que o serviço Autopilot usa para reconhecer o dispositivo.

Como o hash chega ao Intune?

  • OEM/Fornecedor: Dell, HP, Lenovo, Microsoft Surface — todos podem registar o hash diretamente no teu tenant durante a compra (via CSP). É a forma mais limpa.
  • Manual upload (CSV): tu próprio recolhes o hash via PowerShell (script Get-WindowsAutopilotInfo) e fazes upload do CSV no Intune. Útil para dispositivos existentes.
  • Configuration Manager: recolhe o hash automaticamente de dispositivos já geridos.
# Script PowerShell para extrair hardware hash
Install-Script Get-WindowsAutopilotInfo
Get-WindowsAutopilotInfo -OutputFile C:\HWID.csv

Os 5 modos de implantação do Autopilot (clássico / v1)

Modo Quem completa o OOBE Entra Joined? Hybrid Joined? Caso de uso
User-driven O utilizador final Sim Sim Padrão. Utilizador recebe o portátil em casa.
Self-deploying Ninguém (sem utilizador) Sim Não Kiosks, quiosques, salas de reunião, sinaléticas digitais
Pre-provisioned (White Glove) Técnico faz a maior parte; utilizador termina Sim Sim Acelera a entrega: téc prepara, utilizador só faz login final
Autopilot for Existing Devices Existente (passa por wipe-and-load via ConfigMgr) Sim Sim Converter dispositivos antigos para Autopilot
Autopilot Reset Utilizador (após reset) Mantém estado Mantém estado Reciclar dispositivo para novo utilizador rapidamente
⚠️ Restrições importantes do Self-deploying
  • Exige TPM 2.0 com attestation – máquinas virtuais não funcionam
  • Só suporta Entra Joined (não Hybrid Joined)
  • Não há utilizador atribuído ao dispositivo – políticas user-based não se aplicam
  • Não há BitLocker self-service recovery para o utilizador (sem primary user)

Autopilot v2 – Device Preparation (nova geração)

Em 2024 a Microsoft lançou uma nova versão simplificada: Autopilot Device Preparation, frequentemente chamada Autopilot v2. Diferenças principais:

  • ⭐ Não precisa de pré-registar hardware hash! Identifica o dispositivo dinamicamente pelo sign-in
  • Suporta apenas User-driven e Automatic (para Windows 365)
  • Suporta apenas Entra Joined (sem Hybrid Joined)
  • Reporting em near real-time
  • Mais simples, mas menos flexível

Para a prova, é importante saberes que existem os dois: o "clássico" (v1) com todos os modos, e o "novo" (v2) simplificado. Se a questão menciona self-deploying, pre-provisioned, hybrid join ou hardware hash, é v1. Se menciona sem hardware hash, cloud-native e simplicidade, é v2.

Enrollment Status Page (ESP)

É a página que o utilizador vê durante o setup do Autopilot. Mostra o progresso e bloqueia o uso do dispositivo até políticas e apps obrigatórias estarem instaladas.

Definições importantes do ESP:

  • Show app and profile configuration progress: mostra ou esconde o ESP
  • Block device use until all apps and profiles are installed: impede o uso até a instalação terminar
  • Block device use until these required apps are installed: permite escolher selected em vez de all – recomendado!
  • Allow users to reset device if installation error occurs: dá ao utilizador a opção de "começar de novo"
  • Show error when installation takes longer than: timeout (default 60 min)
  • Install Windows quality updates: a partir de janeiro de 2026, instala patches de segurança durante o OOBE
💡 Best practice ESP

Não escolhas "Block device use until ALL apps are installed". Em vez disso, usa "Selected" e identifica explicitamente as apps críticas. Isto evita que o setup falhe por causa de uma app secundária e dá uma experiência mais fluida.

Deployment Profile – campos principais

  • Deployment mode: User-driven ou Self-deploying
  • Join to Microsoft Entra ID as: Microsoft Entra joined ou Microsoft Entra hybrid joined
  • Skip privacy settings, EULA, account setup
  • User account type: Standard (recomendado) ou Administrator
  • Apply device name template: ex. LAB-%RAND:5% nomeia "LAB-A3F9B"
  • Allow pre-provisioned deployment: Yes/No
  • Language (Region): só funciona com Ethernet (Wi-Fi precisa de input do utilizador)
🔬 Lab Prático

Monta o teu próprio ambiente Autopilot

Vais construir um lab completo de Windows Autopilot user-driven, do zero, usando um Microsoft 365 Developer tenant gratuito e uma máquina virtual. Estima 4-6 horas no total, podes dividir por sessões. Este lab vai consolidar uns 30-40% do que vai aparecer na prova.

Passo 1. Criar Microsoft 365 Developer tenant

Vai a developer.microsoft.com/microsoft-365/dev-program e regista-te. Receberás 25 licenças E5 gratuitas (Intune incluído!) renováveis enquanto usares.

Passo 2. Ativar Automatic MDM enrollment

Vai a Entra admin center → Mobility (MDM and WIP) → Microsoft Intune. Define MDM user scope para All. Sem isto, o dispositivo junta-se ao Entra mas nunca se inscreve no Intune – é o erro #1 mais comum.

Passo 3. Criar uma Windows 11 VM no Hyper-V (ou VMware/VirtualBox)

Configurações: 2 vCPU, 4 GB RAM, 60 GB disco, Generation 2 (TPM ativado), Secure Boot. Importante: instala Windows 11 mas não termines o OOBE. Quando aparecer o ecrã de região, faz Ctrl+Shift+F3 ou Shift+F10 para abrir cmd, ou interrompe a VM aqui.

Passo 4. Extrair o hardware hash

Na VM, abre PowerShell como administrator (via Shift+F10 no OOBE) e executa:

md C:\HWID
Set-Location C:\HWID
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Unrestricted -Force
Install-Script -Name Get-WindowsAutopilotInfo
Get-WindowsAutopilotInfo -OutputFile AutopilotHWID.csv

Copia o CSV para o teu computador (rede partilhada, USB, ou OneDrive).

Passo 5. Importar o hash no Intune

Intune admin center → Devices → Windows → Windows enrollment → Devices → Import. Faz upload do CSV. Demora 10-15 minutos a aparecer.

Passo 6. Criar um grupo dinâmico para dispositivos Autopilot

Entra admin center → Groups → New group → Security → Dynamic device. Regra:

(device.devicePhysicalIds -any (_ -contains "[ZTDId]"))

Este grupo passa a conter automaticamente todos os dispositivos Autopilot registados.

Passo 7. Criar o Deployment Profile

Intune → Devices → Windows enrollment → Deployment Profiles → Create profile → Windows PC. Configurações sugeridas:

  • Name: LAB-UserDriven-EntraJoin
  • Deployment mode: User-Driven
  • Join to Entra ID as: Microsoft Entra joined
  • User account type: Standard
  • Skip privacy settings: Yes
  • Skip EULA: Yes
  • Apply device name template: LAB-%RAND:5%
  • Assign to: o grupo dinâmico criado no Passo 6
Passo 8. Criar uma Enrollment Status Page (ESP)

Intune → Devices → Enrollment → Enrollment Status Page → Create. Configurações sugeridas:

  • Show app and profile configuration progress: Yes
  • Block device use until all apps and profiles are installed: Yes
  • Allow users to reset device if installation error occurs: Yes
  • Turn on log collection and diagnostics page: Yes
  • Assign to: o mesmo grupo dinâmico
Passo 9. Adicionar uma app obrigatória (opcional mas recomendado)

Intune → Apps → Windows → Add → Microsoft 365 apps for Windows 10/11. Atribui como Required ao grupo dinâmico. Isto valida que o ESP funciona corretamente.

Passo 10. Executar o OOBE na VM

Volta à VM Windows 11 (reverte ao snapshot pre-OOBE ou recria). Quando aparecer o ecrã "Is this the right country/region?":

  1. Conecta-te à rede (Wi-Fi ou Ethernet)
  2. O Autopilot vai detetar o dispositivo automaticamente e mostrar o nome da tua organização
  3. Faz login com a conta Entra ID do teu tenant developer
  4. Observa o ESP a aplicar políticas e apps
  5. No fim, o dispositivo deve estar Entra Joined, Intune-enrolled, e com o Office instalado
Passo 11. (Bónus) Testar Autopilot Reset

Depois do dispositivo estar pronto, volta ao Intune → Devices → escolhe o dispositivo → Autopilot Reset. O dispositivo vai voltar ao OOBE com tudo apagado, mas mantém-se registado no Autopilot. Excelente para reciclar para outro utilizador.

Notas finais do lab: Self-deploying mode NÃO funciona em VMs porque exige TPM device attestation (que VMs não têm, mesmo com vTPM). Pre-provisioning também precisa de TPM físico. User-driven é o único que dá para testar a 100% em lab virtual.

Microsoft 365 Apps O pacote Office (Word, Excel, etc.) com configurações específicas Windows / macOS
Web link Cria um ícone no dispositivo que abre uma URL Várias

Win32 Content Prep Tool

Ferramenta gratuita que converte ficheiros de instalação (.exe, .msi, ficheiros de configuração) em pacotes .intunewin. Comando básico:

IntuneWinAppUtil.exe -c <source folder> -s <setup file> -o <output folder>

4.2 · Assignments (Atribuições)

Quando atribuis uma app a um grupo, escolhes o intent:

  • Required: instala automaticamente, sem opção do utilizador
  • Available for enrolled devices: aparece no Company Portal para instalação opcional
  • Uninstall: remove a app de dispositivos que a tenham
  • Available with or without enrollment: para apps em dispositivos sem MDM (apenas iOS/Android)

4.3 · App Protection Policies (APP / MAM)

São políticas que protegem dados corporativos dentro de apps, mesmo em dispositivos não geridos. Exemplos do que controlam:

  • Bloquear copiar/colar entre apps corporativas e pessoais
  • Exigir PIN para abrir apps corporativas
  • Cifrar dados em repouso dentro da app
  • Impedir backup para iCloud / Google Drive pessoal
  • Bloquear capturas de ecrã
  • Fazer "selective wipe" remoto apenas dos dados da app
MDM vs MAM

MDM (Mobile Device Management): controlo do dispositivo inteiro. Exige enrollment completo.

MAM (Mobile Application Management): controlo só das apps específicas. Funciona com ou sem enrollment.

BYOD geralmente usa MAM (sem MDM). Dispositivos corporativos geralmente usam MDM + MAM em conjunto.

4.4 · App Configuration Policies

Permitem pré-configurar apps com definições. Exemplo: configurar automaticamente a Outlook app no iOS com a conta corporativa, ou desativar guardar passwords no Edge. Funciona via:

  • App configuration delivered through MDM (precisa de dispositivo enrolled)
  • App configuration delivered through APP (funciona sem enrollment, via MAM)

4.5 · Microsoft 365 Apps for Enterprise

O Office empresarial. No Intune escolhes:

  • Update channel: Current Channel, Monthly Enterprise Channel, Semi-Annual Enterprise Channel
  • Architecture: 64-bit (recomendado) ou 32-bit
  • Apps incluídas: Word, Excel, PowerPoint, Outlook, etc. (escolhes individualmente)
  • Idioma de instalação
  • Activation: Shared computer activation para dispositivos partilhados

10 questões cobrindo deployment de apps, MAM, protection policies. Clica para responder.

Questão 1

Tens um ficheiro setup.exe de uma aplicação comercial e precisas de o implantar via Intune para 500 PCs Windows. Qual é o processo correto?

  1. A. Upload direto do .exe no Intune
  2. B. Usar o Win32 Content Prep Tool para gerar um .intunewin e fazer upload
  3. C. Converter para MSI primeiro
  4. D. Publicar na Microsoft Store
✓ Resposta: B Para fazer upload de qualquer .exe ou .msi como app Win32 no Intune, primeiro tens de o empacotar com o IntuneWinAppUtil.exe, gerando um ficheiro .intunewin. Este formato encripta e empacota o instalador para entrega via Intune.
Questão 2

Queres que uma app apareça no Company Portal para o utilizador instalar quando quiser, sem ser obrigatória. Qual intent de assignment usas?

  1. A. Required
  2. B. Available for enrolled devices
  3. C. Uninstall
  4. D. Detected
✓ Resposta: B Available = a app aparece no Company Portal e o utilizador escolhe quando instalar. Required instala automaticamente sem opção. Uninstall remove. Não existe intent chamado "Detected".
Questão 3

Um utilizador BYOD usa o Outlook no iPhone pessoal. Queres impedir que ele copie texto de emails corporativos para apps pessoais (WhatsApp, etc.). Qual é a abordagem correta?

  1. A. Forçar enrollment MDM do iPhone
  2. B. Aplicar uma App Protection Policy ao Outlook restringindo o copy/paste
  3. C. Bloquear o utilizador via Conditional Access
  4. D. Configurar uma Compliance Policy
✓ Resposta: B App Protection Policies (APP/MAM) protegem dados dentro de apps específicas sem precisar de gerir o dispositivo inteiro. Forçar MDM num iPhone pessoal é invasivo e o utilizador provavelmente recusaria. MAM é a solução elegante para BYOD.
Questão 4

A tua empresa quer Office 365 sempre na última versão estável com novas funcionalidades à medida que saem. Qual update channel deves usar?

  1. A. Current Channel
  2. B. Semi-Annual Enterprise Channel
  3. C. Beta Channel
  4. D. Insider Slow
✓ Resposta: A Current Channel entrega novas funcionalidades assim que ficam estáveis (a cada poucas semanas). Monthly Enterprise é similar mas mais previsível (sempre na segunda terça-feira do mês). Semi-Annual é o mais lento, com atualizações duas vezes por ano — usado em ambientes regulados.
Questão 5

Verdadeiro ou falso: App Protection Policies (APP/MAM) só funcionam em dispositivos que estão inscritos no Intune.

  1. A. Verdadeiro
  2. B. Falso – APP funciona com ou sem enrollment, é uma das suas maiores vantagens
✓ Resposta: B APP funciona sem MDM enrollment – essa é a sua grande vantagem para BYOD. Basta o utilizador adicionar a sua conta corporativa numa app compatível (Outlook, Word, Teams, etc.) e a política é aplicada automaticamente à camada da app, sem gerir o dispositivo todo.
Questão 6

Queres pré-configurar a app do Outlook no iOS para que os utilizadores não precisem inserir manualmente o servidor de email. Que tipo de política usas?

  1. A. Compliance Policy
  2. B. App Configuration Policy
  3. C. App Protection Policy
  4. D. Configuration Profile
✓ Resposta: B App Configuration Policies entregam definições para apps específicas (ex. pré-configurar conta de email no Outlook, definir homepage no Edge). App Protection Policies são sobre segurança/proteção de dados. Não confundir os dois.
Questão 7

No Microsoft 365 Apps for Enterprise, o que é "Shared Computer Activation"?

  1. A. Permite usar uma licença em vários PCs do mesmo utilizador
  2. B. Permite vários utilizadores usarem Office no mesmo dispositivo partilhado, cada um ativado com a sua conta
  3. C. Ativação por chave de produto
  4. D. Ativação offline
✓ Resposta: B Shared Computer Activation é desenhada para cenários como Azure Virtual Desktop, RDS, ou kiosks onde vários utilizadores fazem login no mesmo dispositivo. Cada utilizador "empresta" a sua licença temporariamente. Sem isto, instalar Office num PC partilhado violaria o licenciamento.
Questão 8

O Enterprise App Catalog no Intune oferece o quê?

  1. A. Catálogo de apps gratuitas da Microsoft Store
  2. B. Apps comerciais pré-empacotadas (Zoom, Chrome, Firefox, etc.) prontas para deploy sem necessidade de empacotar
  3. C. Apps Win32 personalizadas
  4. D. Apps Android Enterprise
✓ Resposta: B O Enterprise App Catalog é uma coleção curada pela Microsoft de apps comerciais comuns (Zoom, Chrome, Firefox, Adobe Reader, etc.) já empacotadas e prontas a usar. Poupas o tempo de empacotar manualmente com o IntuneWinAppUtil.
Questão 9

Queres remover apenas dados corporativos de uma app, mantendo a app instalada no dispositivo pessoal. Como é que isto se chama?

  1. A. Full wipe
  2. B. Selective wipe (também chamado app wipe)
  3. C. Fresh Start
  4. D. Reset
✓ Resposta: B Selective wipe (ou app wipe) é executado via App Protection Policy e remove apenas os dados corporativos da app, mantendo a app instalada e os dados pessoais. É o equivalente do "Retire" mas a nível de app individual em vez de dispositivo.
Questão 10

Vais implantar uma app via WinGet no Intune. Qual é a maior vantagem desta abordagem?

  1. A. Mais rápido que .intunewin
  2. B. Não é preciso empacotar nem alojar o instalador – o Intune referencia o pacote WinGet diretamente
  3. C. Suporta Hybrid Joined apenas
  4. D. Funciona offline
✓ Resposta: B Com WinGet, basta indicar o WinGet ID da app (ex: Mozilla.Firefox) e o Intune trata de tudo. Sem empacotamento, sem hosting de ficheiros. A app é descarregada do repositório WinGet oficial. Excelente para apps open-source ou de fornecedores populares.
