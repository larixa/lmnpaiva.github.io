Recursos por segurança · O que cada um faz

  • BitLocker: cifra o disco completo. Protege dados se o dispositivo for roubado.
  • AppLocker: controla quais aplicativos podem correr no dispositivo (allowlist/blocklist por publisher, hash, path).
  • Windows Defender Application Control (WDAC): versão mais robusta do AppLocker, baseada em políticas de assinatura digital. Recomendado para zero trust.
  • Credential Guard: usa virtualização para proteger credenciais (NTLM hash, Kerberos ticket) contra ataques de roubo de credenciais como pass-the-hash. Requer TPM 2.0 e Secure Boot.
  • Windows Defender SmartScreen: bloqueia downloads e sites maliciosos (todas as edições têm isto).
  • Windows Sandbox: ambiente isolado descartável para testar apps suspeitas (Pro+).
  • Memory Integrity (HVCI): proteção da memória do kernel via virtualização.
  • Tamper Protection: impede que malware desative o Defender (todas).
⚠️ Truque cobrado

Se a questão pedir AppLocker, WDAC, Credential Guard ou DirectAccess, a resposta correta é sempre Enterprise ou Education. Pro não tem! Esta é uma das diferenciações mais cobradas entre Pro e Enterprise.

Métodos de Ativação do Windows

Existem quatro grandes métodos de ativação que você precisa conhecer:

  • MAK (Multiple Activation Key): Uma chave única que ativa um número limitado de dispositivos. Cada ativação consome uma "unidade" da chave. Usado em ambientes pequenos ou desconectados.
  • KMS (Key Management Service): Servidor on-premises que ativa clientes na rede interna. Os clientes precisam contatar o servidor KMS pelo menos uma vez a cada 180 dias para manterem a ativação.
  • Active Directory-Based Activation (ADBA): A ativação é feita automaticamente quando o dispositivo se junta ao domínio AD. Não precisa de um servidor KMS dedicado.
  • Subscription Activation: O dispositivo "sobe" automaticamente de edição (Pro → Enterprise) quando o usuário faça login com uma conta Entra ID com licença Windows 11 Enterprise E3/E5. É a forma moderna e a mais cobrada na prova.
Conceito-chave

Subscription Activation é o método preferido em ambientes cloud-first. O dispositivo vem com Windows 11 Pro de fábrica, e ao entrar com a conta Entra ID licenciada, é automaticamente promovido a Enterprise sem necessidade de imagem ou chave de produto. Quando o usuário sai ou perde a licença, o dispositivo volta para Pro.

1.2 · Métodos de Implantação do Windows

Existem várias formas de instalar o Windows num dispositivo. Vou apresentá-las da mais moderna (cloud-first) à mais tradicional (on-premises):

Windows Autopilot (cloud-first, moderno)

Permite que dispositivos novos saiam direto do fornecedor para o usuário final, e que tudo seja configuredo automaticamente quando o usuário liga o dispositivo pela primeira vez. Tem uma seção dedicada mais adiante neste guia.

Provisioning Packages (Windows Configuration Designer)

Arquivos .ppkg criados na ferramenta Windows Configuration Designer. Podem ser aplicados por:

  • Pen USB
  • Email
  • Compartilhamento de rede
  • Diretamente durante o OOBE (Out-of-Box Experience)

Útil para configurer Wi-Fi, certificados, políticas e até para ingressar dispositivos em Entra ID em massa, sem precisar de Intune. Bom para cenários offline ou redes com limitação.

Configuration Manager (anteriormente SCCM)

Solução on-premises da Microsoft para implantação em larga escala. Suporta:

  • Task sequences (sequências de tarefas) para imaginar dispositivos
  • OSD (Operating System Deployment) com PXE boot
  • Integração com Intune via co-management

MDT (Microsoft Deployment Toolkit) e WDS (Windows Deployment Services)

Ferramentas tradicionais para criar e implantar imagens via rede usando PXE boot. Ainda existem mas são consideradas legacy. A Microsoft recomenda fortemente migrar para Autopilot.

1.3 · Caminhos de Upgrade e Feature Updates

Existem três tipos principais de se mover entre versões do Windows:

  • In-place upgrade: Atualizar mantendo apps, arquivos e definições. Suportado de Windows 10 → Windows 11 (se o hardware cumprir os requisitos: TPM 2.0, Secure Boot, CPU compatível).
  • Wipe-and-load: Apaga tudo e instala uma imagem nova. Usado em refresh de hardware antigo.
  • Side-by-side migration: Migração entre dois dispositivos (substituição de hardware).
⚠️ Requisitos do Windows 11

Você precisa conhecer estes requisitos para a prova. Um dispositivo só pode ter Windows 11 se cumprir:

  • TPM 2.0 (Trusted Platform Module)
  • Secure Boot ativado (UEFI)
  • CPU compatível (lista da Microsoft)
  • Mínimo 4 GB RAM e 64 GB de armazenamento
  • DirectX 12 / WDDM 2.0

Se um dispositivo não cumpre, fica no Windows 10 ou tem de ser substituído. Não há "workaround" oficial.

Feature Updates vs Quality Updates

Tipo O que é Frequência Onde se configure
Feature Update Atualização de versão (ex: 23H2 → 24H2). Adiciona novas funcionalidades. Anual Intune → Feature updates for Windows 10 and later
Quality Update Correções de segurança e bugs. O famoso "Patch Tuesday". Mensal Intune → Quality updates ou Update rings
Driver Update Atualizações de drivers via Windows Update for Business Quando disponível Intune → Driver updates

1.4 · Windows 365 e Azure Virtual Desktop

Estes são serviços de Cloud PC – ou seja, em vez do Windows correr no notebook físico, roda em uma máquina virtual na nuvem que o usuário acessa remotamente.

Característica Windows 365 Azure Virtual Desktop (AVD)
Modelo Assinatura fixa por usuário (SaaS) Pay-as-you-go (consumo Azure)
Multi-sessão Não – 1 PC por usuário Sim – vários usuários num host
Gerenciamento Microsoft Intune (mais simples) Portal do Azure (mais complexo, mais flexível)
Caso de uso Trabalhador remoto que precisa de "um PC" na nuvem VDI tradicional, cenários com customização pesada
Como decidir

Se a questão fala em "experiência semelhante a um PC pessoal", "simples de gerenciar" ou "licenciamento previsível por usuário" → Windows 365. Se fala em "vários usuários compartilhando recursos", "otimização de custos" ou "multi-sessão" → Azure Virtual Desktop.

15–20% do exame

2.1 · Os 3 estados de dispositivo no Entra ID

Esta é provavelmente a área mais cobrada da prova MD-102 e onde a maior parte das pessoas se baralha. Vou explicar com calma.

Quando um dispositivo "aparece" no Entra ID, ele pode estar num de três estados:

  1. Microsoft Entra Joined (Entra Joined)
  2. Microsoft Entra Hybrid Joined (Hybrid Joined)
  3. Microsoft Entra Registered (Registered, também chamado Workplace Join)
A ideia central

A diferença entre eles é quem é o "dono" da identidade do dispositivo e que tipo de usuário faça login nele. Vou explicar cada um separadamente, com analogia, e depois fazer uma tabela comparativa.

🔷 Entra Joined

O dispositivo é "cidadão" do Entra ID. Não tem conexão com Active Directory on-premises. O usuário faça login no Windows com a sua conta Entra ID corporativa (nome@empresa.com).

  • Quem se autentica: a conta de trabalho Entra ID
  • Dispositivo é propriedade de: a organização (corporate-owned)
  • Sistema operacional: Windows 10/11 Pro, Enterprise, Education (Home NÃO suporta)
  • Recebe Primary Refresh Token (PRT) no login → SSO automático para Microsoft 365 e apps SaaS
  • Gerenciamento: Microsoft Intune (MDM completo)
  • Casos de uso: ambiente cloud-first, empresas modernas, startups, qualquer organização sem dependência forte de apps legacy on-premises

🔶 Entra Hybrid Joined

O dispositivo está "em duas casas" ao mesmo tempo: ingressado no Active Directory on-premises e também registrado no Entra ID. É o estado da "ponte" para empresas que ainda têm AD on-prem mas querem benefícios do Entra (SSO cloud, Conditional Access, Intune).

  • Quem se autentica: a conta AD (que é sincronizada com Entra ID via Entra Connect ou Entra Cloud Sync)
  • Dispositivo é propriedade de: a organização
  • Sistema operacional: apenas Windows (Windows 10/11 Pro, Enterprise, Education)
  • Precisa de linha de visão para Domain Controller on-premises (problema para remote workers!)
  • Suporta Kerberos/NTLM para apps legacy on-premises (file shares, apps internas)
  • Group Policy (GPO) continua a funcionar
  • Casos de uso: empresas em transição, apps legacy que dependem de AD, file shares Kerberos
⚠️ Recomendação da Microsoft

A Microsoft já não recomenda implantar novos dispositivos como Hybrid Joined. Para novos dispositivos, a recomendação oficial é Entra Joined (cloud-native). Hybrid Joined existe principalmente para suportar quem ainda tem AD legacy. Isto aparece na prova: se a questão pergunta "qual o estado recomendado para NOVOS dispositivos", a resposta é Entra Joined.

🟡 Entra Registered (Workplace Join)

O dispositivo é pessoal – pertence ao usuário, não à organização. O usuário apenas "registra" o dispositivo para conseguir acessar recursos da empresa (email, Teams, etc.) sem perder o controlo do seu próprio dispositivo.

  • Quem se autentica: conta pessoal (Microsoft account local) + uma conta de trabalho adicional para apps específicas
  • Dispositivo é propriedade de: o usuário (BYOD — Bring Your Own Device)
  • Sistema operacional: ⭐ Windows 10/11, macOS, iOS, iPadOS, Android, Linux – é o ÚNICO estado que suporta TUDO
  • Gerenciamento: normalmente MAM (Mobile Application Management), não MDM completo. A organização controla apenas os apps corporativas, não o dispositivo todo.
  • SSO: mais limitado – funciona para apps cloud específicas, não é "transparente" como nos outros estados
  • Casos de uso: BYOD, telefones pessoais a acessar email corporativo, contractors

2.2 · Como adicionar um dispositivo ao Entra ID (na prática)

Vou separar por cenário, porque o caminho que tomas depende muito do tipo de dispositivo. Pensa nisto como uma "árvore de decisão":

Cenário A · Windows novo, corporativo (Entra Joined)

O método depende de quantos dispositivos:

  • Um ou poucos dispositivos: Settings → Accounts → Access work or school → Connect → "Join this device to Microsoft Entra ID". O usuário insira a conta corporativa.
  • Em massa (recomendado): Windows Autopilot — o usuário apenas liga o dispositivo, faça login no OOBE, e tudo é automático. Veja a seção de Autopilot no Domínio 3.
  • Durante o OOBE (Out-of-Box Experience): ao configurer Windows pela primeira vez, escolher "Set up for an organization" e inserir credenciais corporativas → o dispositivo fica automaticamente Entra Joined.

Cenário B · Windows já existente em domínio AD (Hybrid Joined)

O dispositivo já está ingressado no AD on-premises e você quer que também seja registrado no Entra ID. Passos a alto nível:

  1. Instalar e configurer Microsoft Entra Connect (ou Entra Cloud Sync) no servidor on-premises para sincronizar identidades.
  2. Configurar o Service Connection Point (SCP) no AD on-premises a apontar para o tenant Entra.
  3. Configurar uma GPO para "Register domain-joined computers as devices".
  4. Os dispositivos completam o registro automaticamente no próximo login.
💡 Atenção ao Autopilot Hybrid Join

Para Autopilot fazer um dispositivo Hybrid Joined, precisa adicionalmente do Intune Connector for Active Directory instalado num servidor com linha de visão para o domain controller. É fonte comum de problemas e aparece em questões da prova.

Cenário C · iPhone/iPad pessoal a acessar email corporativo (Entra Registered)

O usuário apenas instale o app Microsoft Authenticator ou Company Portal e adiciona a conta corporativa. Não há nada a configurer pelo IT — o dispositivo é automaticamente registrado no Entra. Se houver uma App Protection Policy aplicável, ela é aplicada aos apps relevantes (Outlook, Teams, Word, etc.).

Cenário D · macOS corporativo

macOS pode ser Entra Registered ou totalmente gerenciado via Intune. Para gerenciamento completa:

  1. Configurar a integração com Apple Business Manager (ABM)
  2. Importar tokens MDM da Apple para o Intune
  3. Adicionar números de série dos Macs no ABM
  4. Aplicar uma política Automated Device Enrollment (ADE)
  5. O Mac é configuredo automaticamente no primeiro arranque

2.3 · Como inscrever dispositivos no Intune (na prática)

Atenção: "adicionar ao Entra" e "inscrever no Intune" são duas coisas diferentes, embora muitas vezes aconteçam em conjunto.

Diferença fundamental

Entra ID = identidade do dispositivo (quem ele é, a quem pertence, qual o seu estado).
Intune (MDM) = gerenciamento do dispositivo (políticas, apps, compliance, configurações).

Um dispositivo pode estar no Entra ID sem estar no Intune (ex: dispositivo Entra Registered sem MAM). Mas para gerenciamento completa, normalmente você quer ambos.

Pré-requisito: ativar Automatic Enrollment

Este é o passo mais esquecido (e o que mais causa problemas no lab):

  1. Vai a Entra admin center → Identity → Mobility (MDM and WIP) → Microsoft Intune
  2. Defina o MDM user scope para All (ou um grupo específico)
  3. Mantém o MAM user scope também ativado para BYOD

Sem isto, mesmo que o dispositivo se junte ao Entra, não se inscreve no Intune automaticamente — e o Autopilot vai falhar.

2.4 · O Company Portal · O app central

O Company Portal é o app oficial da Microsoft que serve como "porta de entrada" dos usuários para os recursos corporativos. É uma das ferramentas mais importantes que aparecem na prova — você vai ver questões sobre ela com certeza.

O que faça o Company Portal

  • Inscrever o dispositivo no Intune (em iOS, Android, macOS)
  • Apresentar apps disponíveis (intent "Available") para instalação on-demand pelo usuário
  • Mostrar o estado de compliance do dispositivo ao usuário — "Compliant ✓" ou "Take action"
  • Indicar ao usuário o que fazer para voltar a estar compliant (ex: "Atualize o iOS para 17.0+")
  • Permitir retire/remove do próprio dispositivo (útil para BYOD a sair da empresa)
  • Acessar "Self-service senha reset" e outras funções

2.5 · Autenticação no dispositivo

Windows Hello for Business (WHfB)

Substitui senhas por autenticação biométrica (impressão digital, reconhecimento facial) ou PIN. Importante: o PIN do WHfB é diferente de uma senha — está ligado ao TPM do dispositivo e nunca sai do hardware.

  • Cloud trust model (recomendado): usa Entra ID Kerberos. Não precisa de PKI on-premises.
  • Hybrid certificate trust: usa certificados emitidos por uma CA on-premises.
  • Hybrid key trust: usa chaves armazenadas no TPM, requer configuração de Kerberos.

FIDO2 / Passkeys

Chaves de segurança físicas (USB, NFC) ou passkeys armazenadas em dispositivos. Permitem login senhaless completamente.

Multi-Factor Authentication (MFA)

Combina algo que você sabe (senha) + algo que você tem (celular, token) + algo que você é (biometria). Configurado em Conditional Access ou via Security Defaults.

2.6 · Compliance Policies (Políticas de Conformidade)

Uma Compliance Policy no Intune define quais regras um dispositivo deve cumprir para ser considerado "compliant". Exemplos:

  • Versão mínima do sistema operacional
  • BitLocker ativado
  • Secure Boot ativado
  • Antivirus em dia
  • Sem jailbreak/root
  • Senha com complexidade mínima
  • Microsoft Defender for Endpoint com risk score baixo
Estado do dispositivo

Cada dispositivo é avaliado periodicamente e marcado como Compliant ✅ ou Non-compliant ❌. Este estado é enviado ao Entra ID e pode ser usado em políticas de Conditional Access.

Actions for noncompliance

Quando um dispositivo se torna non-compliant, você pode configurer ações graduais:

  1. Mark device noncompliant imediatamente, ou após X dias de tolerância
  2. Send email to end user a alertar
  3. Send push notification ao usuário
  4. Remotely lock the device
  5. Retire the noncompliant device (remoção completa)

2.7 · Conditional Access (CA)

É o "porteiro" do Entra ID. Uma política de Conditional Access tem dois lados:

Condições (Quando?) Controlos (O quê?)
Usuários ou grupos Bloquear acesso (Block)
Apps ou recursos Exigir MFA
Localização (IP, país) Exigir dispositivo compliant
Plataforma do dispositivo Exigir dispositivo Hybrid Joined
Estado do dispositivo Exigir app aprovada (ex: Outlook)
Risco do sign-in / usuário Exigir mudança de senha
💡 AND vs OR nos controlos

Os controlos de "Grant" podem ser combinados:

  • Require all of the selected controls (AND): "Exigir MFA E dispositivo compliant" — muito mais restritivo
  • Require one of the selected controls (OR): "Exigir MFA OU dispositivo compliant" — menos restritivo

A prova adora testar isto. Leia com atenção qual operador está sendo usado.

Report-only mode

Antes de "ligar" uma política em produção, você pode pô-la em Report-only. A política é avaliada para cada sign-in mas não é aplicada – só registra no log o que teria acontecido. Excelente para testar sem partir nada.

2.8 · Intune RBAC e Scope Tags

Em organizações grandes, você quer separar quem pode gerenciar o quê. O Intune tem três conceitos:

  • Roles: "School Administrator", "Help Desk Operator", "Policy and Profile Manager", etc.
  • Scope Tags: etiquetas que limitam quais recursos um admin veja. Ex: scope tag "Lisbon" só veja dispositivos do escritório de Lisboa.
  • Assignments: a quem o role/scope é atribuído.

🔍 Android Enterprise · Mergulho profundo

Android é a plataforma mais complexa do Intune porque tem 4 cenários completamente diferentes, cada um com o seu próprio fluxo de enrollment, capacidades de gerenciamento e casos de uso. Vou explicar cada um com calma.

Pré-requisito: conexão Google Play

Antes de qualquer enrollment Android, você precisa fazer uma conexão única: Intune admin center → Devices → Android → Android Enrollment → Managed Google Play. Ligas a sua conta Google empresarial e aprovas a integração com o Intune. Sem isto, nenhum dos 4 modos funciona.

Modo 1 · Android Enterprise Fully Managed

O dispositivo é 100% da empresa. Não há espaço pessoal. Tudo o que o usuário faça é gerenciado pela organização. Equivalente ao "supervised" do iOS.

  • Quem é dono: Organização
  • Espaço pessoal: ❌ Não existe
  • Como inscrever: Factory reset → durante setup wizard, no tela de email, digita afw#setup (ou QR code do Intune ou NFC tap)
  • Apps: Apenas Managed Google Play (apps aprovadas pelo admin)
  • Caso de uso: Celulares corporativos de comerciais, técnicos no terreno, dispositivos atribuídos individualmente

Modo 2 · Android Enterprise Dedicated

O dispositivo é de uso único, sem usuário atribuído. Funciona como um kiosk — fica bloqueado a uma ou poucas apps. O usuário (cliente, funcionário em rotação) não faça login.

  • Quem é dono: Organização
  • Usuário atribuído: ❌ Não há (anónimo)
  • Como inscrever: Factory reset → QR code do Intune (ou NFC, Zero-Touch)
  • Modo kiosk: Sim — pode bloquear a uma único app (single-app) ou poucos apps (multi-app)
  • Caso de uso: Tablets em montra de loja, terminais de inventário em armazém, leitores de códigos, sistemas de checkout, máquinas de pedidos no restaurante

Modo 3 · Corporate-Owned Work Profile (COPE)

O dispositivo é da empresa, mas a empresa permite que o usuário também o use pessoalmente. Tem dois "perfis" no mesmo dispositivo, completamente separados.

  • Quem é dono: Organização (mas com uso pessoal permitido)
  • Espaço pessoal: ✅ Separado do profissional (apps, dados, fotos próprias do usuário)
  • Como inscrever: Factory reset → durante setup wizard, escolher "Set up work profile" ou via Zero-Touch / KME
  • O que o admin vê: Apenas o lado profissional (privacy guaranteed para o pessoal)
  • O que o admin pode fazer: Wipe ao dispositivo todo (admin escolhe) ou apenas ao work profile
  • Caso de uso: Empresas que dão celular mas reconhecem que vai ser usado pessoalmente também — gestores, comerciais, dirigentes

Modo 4 · Personally-Owned Work Profile (BYOD)

O dispositivo é do usuário (BYOD). A empresa apenas cria um "work profile" isolado dentro do celular pessoal. Tudo o resto é território pessoal — a empresa nem veja.

  • Quem é dono: Usuário
  • Espaço pessoal: ✅ Predominante; o trabalho está num "compartimento" isolado
  • Como inscrever: Usuário instalo app Intune Company Portal da Play Store → adiciona conta corporativa → aceita criação do work profile
  • Apps profissionais: Aparecem com um pequeno ícone de mala (badge) para identificar visualmente
  • Privacy: O admin NÃO vê apps pessoais, fotos, contatos, browsing
  • Wipe: Apenas o work profile pode ser apagado (retire); o lado pessoal permanece intacto
  • Caso de uso: Funcionários a usarem o celular pessoal para email/Teams corporativos

Métodos de inscrição em massa para Android

  • Zero-Touch Enrollment (ZTE): protocolo do Google. O retailer/distribuidor associa o dispositivo ao tenant; o usuário apenas liga e o setup é automático. Não precisa de QR codes nem touch físico.
  • Samsung Knox Mobile Enrollment (KME): equivalente da Samsung, específico para dispositivos Samsung. Funciona de forma parecida ao Zero-Touch.
  • QR code enrollment: técnico faça factory reset, abre o setup wizard, e usa a câmera para ler um QR code gerado pelo Intune. Útil para volumes pequenos.
  • afw#setup: escreves afw#setup no tela de email durante o setup wizard. O Android baixa automaticamente o Android Device Policy app.
  • NFC enrollment: tap-to-enroll entre dois dispositivos. Hoje raramente usado.
⚠️ Importante para a prova

Para enrollar Fully Managed, Dedicated ou COPE, o dispositivo tem de estar numa fábrica fresh state (factory reset). Não dá para converter um dispositivo já em uso pessoal para esses modos sem reset. Só o Personally-Owned Work Profile pode ser adicionado a um dispositivo já em uso.

3.4 · Microsoft Defender for Endpoint (MDE)

É a plataforma EDR (Endpoint Detection and Response) da Microsoft. Vai muito além do antivírus tradicional. Componentes principais:

  • Next-gen antivirus: proteção em tempo real, cloud-delivered protection
  • EDR (Endpoint Detection and Response): deteta comportamentos suspeitos, não apenas malware conhecido
  • Attack Surface Reduction (ASR) rules: bloqueia técnicas comuns de ataque (macros, scripts, ransomware behaviour)
  • Threat & Vulnerability Management (TVM): identifique vulnerabilidades nos dispositivos
  • Automated Investigation and Remediation (AIR): resposta automática a incidentes

Onboarding de dispositivos no Defender

  • Via Intune: Configurar a "Endpoint detection and response" policy e atribuar aos grupos. É a forma recomendada.
  • Via Group Policy: para Hybrid Joined
  • Via script local: para dispositivos isolados
  • Via Configuration Manager

Conditional Access baseado em risco

O MDE atribua um Machine Risk Score a cada dispositivo (Low, Medium, High). Você pode integrar isto numa Compliance Policy para considerar non-compliant um dispositivo com risk score "Medium" ou superior. Isto, combinado com Conditional Access, cria um ciclo de defesa automático.

3.5 · Windows Autopatch e Update Management

Update Rings

Permite definir anéis de implementação graduais para feature e quality updates:

  1. Pilot ring: 5-10 usuários. Recebem updates primeiro.
  2. Early adopters: ~20% do parque. Recebem 1 semana depois.
  3. Broad deployment: ~70% do parque. Recebem 2 semanas depois.
  4. Critical (last): dispositivos críticos. Recebem por último.

3.6 · BitLocker e Encryption

Cifra o disco completo, protegendo os dados se o dispositivo for roubado ou perdido.

  • TPM only: o dispositivo desbloqueia automaticamente no boot (sem PIN). Simples mas menos seguro contra ataques físicos.
  • TPM + PIN: o usuário insira um PIN no boot. Mais seguro.
  • Recovery key: chave de 48 dígitos que pode desbloquear o dispositivo. É guardada automaticamente no Entra ID quando o dispositivo é Entra Joined ou Hybrid Joined.
Onde encontrar a recovery key

Para Entra Joined: Entra admin center → Devices → escolher o dispositivo → "BitLocker keys". Para Hybrid Joined: pode estar guardada no AD on-premises ou no Entra (depende da configuração).

3.9 · Windows Autopilot · Mergulho Profundo

Esta seção é dedicada porque pediste detalhamento profundo, e porque Autopilot aparece em quase todas as provas MD-102. Vou explicar tudo: o que é, como funciona internamente, todos os modos, o que cada um exige, e no fim você tem um lab para montar.

O que é o Windows Autopilot

Imagina o cenário tradicional: um técnico de IT recebe um notebook novo, instala uma imagem do Windows (golden image), configure o domínio, instala apps, configure políticas, e só depois o envia para o usuário. Demora horas e exige um técnico físico.

O Autopilot vira esta história ao contrário: o notebook vai direto do fornecedor para o usuário final. Quando o usuário liga o notebook pela primeira vez, se conecta à internet e insira as credenciais Entra ID, o Autopilot:

  1. Reconhece o dispositivo pelo seu hardware hash (já pré-registrado no serviço)
  2. Aplica um perfil de Autopilot com configurações de OOBE
  3. Junta o dispositivo ao Entra ID
  4. Inscreve o dispositivo automaticamente no Intune
  5. Instala apps obrigatórias, políticas, certificados, BitLocker, etc.
  6. Entrega o dispositivo "pronto a usar" diretamente ao usuário

Hardware Hash – a "impressão digital" do dispositivo

Cada dispositivo Windows tem uma identificação única gerada a partir do hardware (CPU, motherboard, TPM, MAC address). Este "hash" é o que o serviço Autopilot usa para reconhecer o dispositivo.

Como o hash chega ao Intune?

  • OEM/Fornecedor: Dell, HP, Lenovo, Microsoft Surface — todos podem registrar o hash diretamente no seu tenant durante a compra (via CSP). É a forma mais limpa.
  • Manual upload (CSV): você mesmo extrai o hash via PowerShell (script Get-WindowsAutopilotInfo) e faz upload do CSV no Intune. Útil para dispositivos existentes.
  • Configuration Manager: recolhe o hash automaticamente de dispositivos já gerenciados.
# Script PowerShell para extrair hardware hash
Install-Script Get-WindowsAutopilotInfo
Get-WindowsAutopilotInfo -OutputFile C:\HWID.csv

⚠️ Restrições importantes do Self-deploying
  • Exige TPM 2.0 com attestation – máquinas virtuais não funcionam
  • Só suporta Entra Joined (não Hybrid Joined)
  • Não há usuário atribuído ao dispositivo – políticas user-based não se aplicam
  • Não há BitLocker self-service recovery para o usuário (sem primary user)

Autopilot v2 – Device Preparation (nova geração)

Em 2024 a Microsoft lançou uma nova versão simplificada: Autopilot Device Preparation, frequentemente chamada Autopilot v2. Diferenças principais:

  • ⭐ Não precisa de pré-registrar hardware hash! Identifique o dispositivo dinamicamente pelo sign-in
  • Suporta apenas User-driven e Automatic (para Windows 365)
  • Suporta apenas Entra Joined (sem Hybrid Joined)
  • Reporting em near real-time
  • Mais simples, mas menos flexível

Para a prova, é importante saberes que existem os dois: o "clássico" (v1) com todos os modos, e o "novo" (v2) simplificado. Se a questão menciona self-deploying, pre-provisioned, hybrid join ou hardware hash, é v1. Se menciona sem hardware hash, cloud-native e simplicidade, é v2.

Enrollment Status Page (ESP)

É a página que o usuário veja durante o setup do Autopilot. Mostra o progresso e bloqueia o uso do dispositivo até políticas e apps obrigatórias estarem instaladas.

Definições importantes do ESP:

  • Show app and profile configuretion progress: mostra ou esconde o ESP
  • Block device use until all apps and profiles are installed: impede o uso até a instalação terminar
  • Block device use until these required apps are installed: permite escolher selected em vez de all – recomendado!
  • Allow users to reset device if installation error occurs: dá ao usuário a opção de "começar de novo"
  • Show error when installation takes longer than: timeout (default 60 min)
  • Install Windows quality updates: a partir de janeiro de 2026, instala patches de segurança durante o OOBE
💡 Best practice ESP

Não escolhas "Block device use until ALL apps are installed". Em vez disso, usa "Selected" e identifique explicitamente os apps críticas. Isto evita que o setup falhe por causa de umo app secundária e dá uma experiência mais fluida.

Deployment Profile – campos principais

  • Deployment mode: User-driven ou Self-deploying
  • Join to Microsoft Entra ID as: Microsoft Entra joined ou Microsoft Entra hybrid joined
  • Skip privacy settings, EULA, account setup
  • User account type: Standard (recomendado) ou Administrator
  • Apply device name template: ex. LAB-%RAND:5% nomeia "LAB-A3F9B"
  • Allow pre-provisioned deployment: Yes/No
  • Language (Region): só funciona com Ethernet (Wi-Fi precisa de input do usuário)
🔬 Lab Prático

Monta o seu próprio ambiente Autopilot

Você vai construir um lab completo de Windows Autopilot user-driven, do zero, usando um Microsoft 365 Developer tenant gratuito e uma máquina virtual. Estima 4-6 horas no total, você pode dividir por sessões. Este lab vai consolidar uns 30-40% do que vai aparecer na prova.

Passo 1. Criar Microsoft 365 Developer tenant

Vai a developer.microsoft.com/microsoft-365/dev-program e registre-se. Receberás 25 licenças E5 gratuitas (Intune incluído!) renováveis enquanto usares.

Passo 2. Ativar Automatic MDM enrollment

Vai a Entra admin center → Mobility (MDM and WIP) → Microsoft Intune. Defina MDM user scope para All. Sem isto, o dispositivo se associa ao Entra mas nunca se inscreve no Intune – é o erro #1 mais comum.

Passo 3. Criar uma Windows 11 VM no Hyper-V (ou VMware/VirtualBox)

Configurações: 2 vCPU, 4 GB RAM, 60 GB disco, Generation 2 (TPM ativado), Secure Boot. Importante: instala Windows 11 mas não termines o OOBE. Quando aparecer o tela de região, faça Ctrl+Shift+F3 ou Shift+F10 para abrir cmd, ou interrompe a VM aqui.

Passo 4. Extrair o hardware hash

Na VM, abre PowerShell como administrator (via Shift+F10 no OOBE) e executa:

md C:\HWID
Set-Location C:\HWID
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Unrestricted -Force
Install-Script -Name Get-WindowsAutopilotInfo
Get-WindowsAutopilotInfo -OutputFile AutopilotHWID.csv

Copie o CSV para o seu computador (rede compartilhada, USB, ou OneDrive).

Passo 5. Importar o hash no Intune

Intune admin center → Devices → Windows → Windows enrollment → Devices → Import. Faça upload do CSV. Demora 10-15 minutos a aparecer.

Passo 6. Criar um grupo dinâmico para dispositivos Autopilot

Entra admin center → Groups → New group → Security → Dynamic device. Regra:

(device.devicePhysicalIds -any (_ -contains "[ZTDId]"))

Este grupo passa a conter automaticamente todos os dispositivos Autopilot registrados.

Passo 7. Criar o Deployment Profile

Intune → Devices → Windows enrollment → Deployment Profiles → Create profile → Windows PC. Configurações sugerenciadas:

  • Name: LAB-UserDriven-EntraJoin
  • Deployment mode: User-Driven
  • Join to Entra ID as: Microsoft Entra joined
  • User account type: Standard
  • Skip privacy settings: Yes
  • Skip EULA: Yes
  • Apply device name template: LAB-%RAND:5%
  • Assign to: o grupo dinâmico criado no Passo 6
Passo 8. Criar uma Enrollment Status Page (ESP)

Intune → Devices → Enrollment → Enrollment Status Page → Create. Configurações sugerenciadas:

  • Show app and profile configuretion progress: Yes
  • Block device use until all apps and profiles are installed: Yes
  • Allow users to reset device if installation error occurs: Yes
  • Turn on log collection and diagnostics page: Yes
  • Assign to: o mesmo grupo dinâmico
Passo 9. Adicionar umo app obrigatória (opcional mas recomendado)

Intune → Apps → Windows → Add → Microsoft 365 apps for Windows 10/11. Atribua como Required ao grupo dinâmico. Isto valida que o ESP funciona corretamente.

Passo 10. Executar o OOBE na VM

Volta à VM Windows 11 (reverte ao snapshot pre-OOBE ou recria). Quando aparecer o tela "Is this the right country/region?":

  1. Conecte-se à rede (Wi-Fi ou Ethernet)
  2. O Autopilot vai detetar o dispositivo automaticamente e mostrar o nome da sua organização
  3. Faça login com a conta Entra ID do seu tenant developer
  4. Observa o ESP a aplicar políticas e apps
  5. No fim, o dispositivo deve estar Entra Joined, Intune-enrolled, e com o Office instalado
Passo 11. (Bónus) Tester Autopilot Reset

Depois do dispositivo estar pronto, volta ao Intune → Devices → escolha o dispositivo → Autopilot Reset. O dispositivo vai voltar ao OOBE com tudo apagado, mas se mantém registrado no Autopilot. Excelente para reciclar para outro usuário.

Notas finais do lab: Self-deploying mode NÃO funciona em VMs porque exige TPM device attestation (que VMs não têm, mesmo com vTPM). Pre-provisioning também precisa de TPM físico. User-driven é o único que dá para testar a 100% em lab virtual.

Win32 Content Prep Tool

Ferramenta gratuita que converte arquivos de instalação (.exe, .msi, arquivos de configuração) em pacotes .intunewin. Comando básico:

IntuneWinAppUtil.exe -c <source folder> -s <setup file> -o <output folder>

4.2 · Assignments (Atribuações)

Quando atribuas umo app a um grupo, você escolha o intent:

  • Required: instala automaticamente, sem opção do usuário
  • Available for enrolled devices: aparece no Company Portal para instalação opcional
  • Uninstall: remove o app de dispositivos que a tenham
  • Available with or without enrollment: para apps em dispositivos sem MDM (apenas iOS/Android)

4.3 · App Protection Policies (APP / MAM)

São políticas que protegem dados corporativos dentro de apps, mesmo em dispositivos não gerenciados. Exemplos do que controlam:

  • Bloquear copiar/colar entre apps corporativas e pessoais
  • Exigir PIN para abrir apps corporativas
  • Cifrar dados em repouso dentro da app
  • Impedir backup para iCloud / Google Drive pessoal
  • Bloquear capturas de tela
  • Fazer "selective wipe" remoto apenas dos dados da app
MDM vs MAM

MDM (Mobile Device Management): controlo do dispositivo inteiro. Exige enrollment completo.

MAM (Mobile Application Management): controlo só das apps específicas. Funciona com ou sem enrollment.

BYOD geralmente usa MAM (sem MDM). Dispositivos corporativos geralmente usam MDM + MAM em conjunto.

4.4 · App Configuration Policies

Permitem pré-configurar apps com definições. Exemplo: configurer automaticamente a Outlook app no iOS com a conta corporativa, ou desativar guardar senhas no Edge. Funciona via:

  • App configuretion delivered through MDM (precisa de dispositivo enrolled)
  • App configuretion delivered through APP (funciona sem enrollment, via MAM)

4.5 · Microsoft 365 Apps for Enterprise

O Office empresarial. No Intune você escolhe:

  • Update channel: Current Channel, Monthly Enterprise Channel, Semi-Annual Enterprise Channel
  • Architecture: 64-bit (recomendado) ou 32-bit
  • Apps incluídas: Word, Excel, PowerPoint, Outlook, etc. (você escolha individualmente)
  • Idioma de instalação
  • Activation: Shared computer activation para dispositivos comcompartilhados
